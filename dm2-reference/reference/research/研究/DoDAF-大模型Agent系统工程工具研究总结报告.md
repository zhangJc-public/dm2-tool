# DoDAF × 大模型 Agent 系统工程工具研究总结报告

> ⚠️ **本文档为前瞻性研究，非 DoDAF 2.02 标准内容。**
>
> **研究目的**：总结将 DoDAF / DM2 与大模型 Agent、Obsidian、Claude Code 结合，形成一套可实施系统工程工具链的关键认识、正确思路与落地路径。
>
> **研究结论（一句话）**：DoDAF × 大模型 Agent 的关键不在于“多智能体生成更多视图”，而在于建立一套**以 SSOT 为核心、以产物生命周期为主轴、以评审门驱动收敛**的系统工程生产线。

---

## 一、研究背景与问题界定

在实践中，利用 Skills、Agent 或 Claude Code 生成 DoDAF 各类视图并不困难，困难在于：

1. **输出可看但不可用**：视图看似完整，但术语漂移、对象定义不稳、跨视图关系无法追溯。
2. **输出可生成但不可维护**：每次重生成都可能改变表述、结构和粒度，难以形成稳定基线。
3. **输出可描述但不可审查**：缺少统一事实源与明确验收门槛，评审只能停留在“感觉不满意”。
4. **方法被实例绑架**：一旦从某个设备、某个产品或某个项目实例出发，抽象层就容易坍塌，最终失去可复用性。

因此，真正的问题不是“如何让大模型生成更多视图”，而是：

- 如何让 DoDAF 视图建立在**稳定、可追溯、可审查**的事实基础之上；
- 如何让 LLM / Agent 从“文本生成器”升级为“知识加工器、校验器、修补建议器”；
- 如何让系统工程过程围绕**产物状态与评审门**收敛，而不是围绕 prompt 自由发挥。

---

## 二、核心判断：DoDAF 不是视图生成问题，而是多视角一致性问题

本次研究最重要的共识是：

### 2.1 DoDAF 的本质
DoDAF 不应被理解为“若干视图模板”，而应理解为：

**通过多个视角对同一套架构事实进行约束、投影和验证的体系。**

因此，系统的重点不应该放在：
- 如何一次性生成更多 Markdown 视图；

而应该放在：
- 如何维护同一套事实；
- 如何让不同视图从同一事实派生；
- 如何通过视图之间的关系发现冲突、缺失与伪一致性。

### 2.2 大模型的正确角色
大模型不应直接担任“最终文档撰写器”，其更合理的职责是：

- **抽取**：从原始资料中抽取实体、关系、约束与证据；
- **对齐**：完成同义归并、术语标准化、关系补全；
- **派生**：从事实基线生成视图骨架、表格、草图与 Markdown 文本；
- **校验**：执行跨视角一致性检查，形成问题单与修补建议；
- **辅助评审**：汇总异议、解释差异、给出回写建议，但不替代人工裁决。

这意味着：

**LLM 是生产线中的智能工位，不是真相源，更不是最终裁决者。**

---

## 三、导致“生成视图但不满意”的根因分析

结合实践经验，已有方案不理想通常来自以下几类根因。

### 3.1 缺少 SSOT（Single Source of Truth，单一事实来源）
如果没有统一词表、统一对象 ID、统一关系表，那么每个 Agent 都会隐含地维护自己的“局部真相”，最终导致：

- 同一对象多名称；
- 同一能力多定义；
- 同一交换关系在 OV、SV、DIV 中口径不一致；
- 后续评审无法判断问题出在事实层还是表达层。

### 3.2 把“视图文档”当成主产物
如果视图文本本身就是主对象，那么每次修改都会落在文档层，无法沉淀为可重建的基线。结果是：

- 人工修文越来越重；
- 模型重生后结果漂移；
- 没有稳定的追溯链；
- 无法进行可靠的差异分析。

### 3.3 把系统工程误建模成线性流水线
系统工程不是单向步骤，而是高度依赖：

- 评议
- 审核
- 审查
- 评审
- 退回修改
- 再提交
- 再基线化

因此，其过程更接近**产物中心、评审驱动、状态收敛**的协同系统，而不是简单的多 Agent 流程图。

### 3.4 从具体产品实例出发导致抽象层失效
如果一开始就围绕具体设备、具体厂商型号展开，容易出现：

- 视图结构由产品功能决定；
- 抽象概念退化成产品功能清单；
- 一旦换产品，整个模型重写；
- 最终无法形成“参考架构”，只能形成“个案笔记”。

---

## 四、正确思路：抽象优先、基线优先、评审优先

本研究认为，正确路径应遵循以下三条总原则。

### 4.1 抽象优先：先参考模型，再实例绑定
方法论的起点不应是某个设备或产品实例，而应是：

1. **能力（Capability）**：组织需要什么能力；
2. **活动（Operational Activity）**：能力通过什么业务/作业活动体现；
3. **逻辑执行者/逻辑服务（Logical Performer / Service）**：哪些逻辑角色承担责任；
4. **资源实例（Resource Instance）**：最后才映射到具体系统、产品、接口、版本。

一句话概括：

**先做参考体系，再做实例投影。**

### 4.2 基线优先：先修改事实，再重建视图
所有视图都应是派生产物，而不是手工最终稿。任何重要修改都应先落到：

- 术语词表；
- 对象定义；
- 关系追溯；
- 规则与约束；
- 评审裁决记录。

然后再由这些基线重新生成或刷新视图。

### 4.3 评审优先：评审不是异常流程，而是主流程
在系统工程中，评议、审核、审查、评审不应被视为“生成后顺便做一下”，而应成为过程主轴。

因此，统一调度不应只是“谁先执行哪个 Agent”，而应关注：

- 哪个产物处于什么状态；
- 当前进入哪个评审门；
- 哪些问题已裁决，哪些仍开放；
- 哪些视图可以基线化，哪些必须退回重做。

---

## 五、统一调度的正确对象：不是 Agent，而是产物生命周期

### 5.1 为什么不应以 Agent 作为统一调度中心
虽然可以设计多个 Agent（抽取、生成、校验、评审整理等），但如果统一调度建立在 Agent 图上，会有几个问题：

- 过程容易被误简化为固定流程；
- 评审与回退难以自然表达；
- 基线和状态难以长期管理；
- 重心落在“谁生成”而不是“产物是否成熟”。

### 5.2 更合理的统一调度模型
统一调度应围绕以下四类对象展开：

#### 1. 基线对象（Baseline Objects）
- 术语词表（Glossary）
- 元模型/关系表（Model / Trace）
- 规则与约束（Rules）
- 决策记录（Decision Record）

#### 2. 产物对象（Artifacts）
- Capability 草案
- OV / SV / DIV / CV 草案
- 一致性报告
- 评审意见单
- 差异分析单

#### 3. 状态对象（States）
推荐使用以下状态：
- `draft`
- `generated`
- `under_review`
- `rework`
- `approved`
- `baselined`
- `superseded`

#### 4. 评审门（Review Gates）
例如：
- 术语审定门
- 能力-活动映射审定门
- 交换-接口-数据一致性审定门
- 实例绑定审定门

### 5.3 统一调度的一句话定义
**统一调度，不是谁来发 prompt，而是谁来管理“产物状态、评审门、决策记录和基线版本”。**

---

## 六、Claude Code + Obsidian 的合理定位

本研究认为，`Claude Code + Obsidian` 是一条现实且合理的路线，尤其适合当前阶段。

### 6.1 Obsidian 的正确角色
Obsidian 更适合作为：

- **SSOT 承载层**：词表、关系、基线、决策、评审结论；
- **人工评审界面**：人类可以直接阅读、批注、比较和双链引用；
- **知识沉淀空间**：使抽象模型、参考模式和实例绑定长期共存。

因此，Obsidian 不只是笔记库，而应成为：

**架构事实基线与评审记录的知识底座。**

### 6.2 Claude Code 的正确角色
Claude Code 更适合作为：

- 结构化抽取器；
- 关系补全器；
- 视图派生器；
- 一致性检查器；
- 评审意见整理器；
- 差异修复建议器。

Claude Code 不应承担：

- 真相源角色；
- 自主裁决角色；
- 任意修改基线的权限。

### 6.3 组合关系
最佳职责分工可以表述为：

- **Obsidian 负责“存什么”**；
- **Claude Code 负责“怎么改”**；
- **人工架构师负责“什么能过”**。

---

## 七、最小可实施架构（Minimum Viable Architecture）

在当前阶段，不建议一开始就构造复杂多 Agent 平台，而建议采用一个更稳的最小架构。

### 7.1 推荐结构

#### A. 事实基线层（SSOT）
至少包含：
- `glossary`：统一术语、定义、对象 ID、别名；
- `trace`：能力—活动—逻辑角色—服务—交换—数据之间的追溯关系；
- `rules`：术语规范、映射约束、评审门规则；
- `decision-log`：人工裁决与设计决策记录。

#### B. 产物层（Artifacts）
包含：
- 参考能力模型；
- 参考活动链；
- 参考逻辑架构；
- 视图草案；
- 一致性报告；
- 评审问题单。

#### C. 智能加工层（Claude Code）
执行：
- 抽取；
- 对齐；
- 生成；
- 校验；
- 修补建议。

#### D. 人工审查层（Human-in-the-loop）
执行：
- 术语归并；
- 冲突裁决；
- 重要视图放行；
- 基线版本批准。

### 7.2 初期不建议过度追求的内容
- 不建议一开始部署过多专职 Agent；
- 不建议过早引入重量级图式编排；
- 不建议先围绕具体产品或设备细节展开；
- 不建议直接生成“最终版”长文档而缺少结构化中间产物。

---

## 八、最小 SSOT 设计建议

为保证可实施性，SSOT 应保持“足够结构化，但不过度复杂”。

### 8.1 最小必要对象
建议至少维护以下对象类型：

- `Capability`
- `OperationalActivity`
- `LogicalPerformer`
- `Service`
- `InformationObject`
- `DataElement`
- `Constraint`
- `Measure`
- `Decision`
- `InstanceBinding`

### 8.2 三条硬规则

#### 规则 1：任何视图中的名词，必须在词表中有唯一 ID
若某名词未入词表，则该视图不能视为合格。

#### 规则 2：任何视图中的关系，必须可追溯到 trace 或规则
若关系无法回到基线，则它只是“叙述”，不是“架构事实”。

#### 规则 3：任何最终视图都必须可重建
若某视图只能靠人工补丁维持，则说明事实层尚未完成。

---

## 九、视图质量门槛（Definition of Done）建议

为了避免“凭感觉不满意”，每类视图需要明确完成定义。

### 9.1 OV 类视图
至少满足：
- 活动、参与者、交换对象清晰；
- 无未定义术语；
- 关键交换有唯一标识；
- 关键活动能够映射到能力；
- 关键交换能够映射到后续接口或数据对象。

### 9.2 SV 类视图
至少满足：
- 逻辑服务与承担职责明确；
- 与 OV 中的活动/交换存在可说明映射；
- 接口或交互边界可识别；
- 不直接依赖未审定的具体产品细节。

### 9.3 DIV 类视图
至少满足：
- 关键交换对象在数据字典中存在定义；
- 信息对象与数据元素语义一致；
- 来源、用途、关键约束可说明；
- 与 SV 中的交换矩阵或接口说明能够对齐。

### 9.4 跨视图一致性门槛
至少检查：
- OV 中的交换是否在 SV / DIV 落地；
- CV 中的能力是否在 OV 活动中体现；
- 术语是否统一；
- 实例绑定是否只发生在抽象模型稳定之后。

## 十、实施路线：从“方法验证”到“体系化运行”

### 阶段一：最小闭环验证
目标：证明方法成立，而不是追求覆盖全部视图。

建议只做：
- 一个参考场景；
- 一组能力；
- 一条活动链；
- 一套交换对象；
- 一次视图生成与一致性检查。

关键产出：
- 最小词表；
- 最小追溯表；
- 1 轮评审问题单；
- 1 组重建后的视图。

### 阶段二：参考架构化
目标：从个案走向模式。

建议增加：
- 参考能力分类；
- 参考活动模板；
- 参考逻辑角色；
- 参考交换对象分类；
- 参考评审门。

这一阶段的重点是**去产品化、去厂商化、去项目化**。

### 阶段三：实例绑定化
目标：把稳定参考模型映射到具体系统、产品、项目。

建议内容：
- 哪个实例承担哪个逻辑角色；
- 哪个产品承接哪个服务职责；
- 哪个接口落到哪个实现对象；
- 哪些能力因实例替换而受到影响。

### 阶段四：生命周期治理
目标：使系统成为持续运行的工程工具，而不是一次性实验。

建议增加：
- 版本基线管理；
- 评审记录闭环；
- 差异与影响分析；
- 局部重建与局部修复机制。

---

## 十一、反模式清单（应尽量避免）

以下做法会显著降低工具的长期价值：

1. **一开始就试图覆盖全部 DoDAF 视图**；
2. **把具体设备/产品作为方法起点**；
3. **让视图文档直接承载最终真相**；
4. **让多个 Agent 各自维护术语和关系**；
5. **缺少人工评审门，只依赖模型自评**；
6. **把系统工程过程过度简化为固定线性流水线**；
7. **没有基线、没有状态、没有决策记录**。

---

## 十二、研究结论

综合本次讨论，可以形成如下结论：

### 12.1 方法论结论
DoDAF × 大模型 Agent 的成功关键，不在于“模型多强、Agent 多少、编排多复杂”，而在于是否建立了：

- 统一事实源；
- 产物生命周期；
- 评审门；
- 决策与基线管理。

### 12.2 工具路线结论
`Claude Code + Obsidian` 不是权宜之计，而是一条合理路线。其成立前提是：

- Obsidian 从“笔记仓库”升级为“SSOT 与评审底座”；
- Claude Code 从“内容生成器”升级为“知识加工与校验工位”；
- 人工架构师继续保留术语审定、冲突裁决和基线批准权。

### 12.3 实施策略结论
更可行的路径不是“先做多 Agent 编排平台”，而是：

1. 先定义基线对象；
2. 再定义产物类型；
3. 再定义状态迁移；
4. 再定义评审门；
5. 最后再逐步自动化局部智能工位。

### 12.4 一句话最终结论
**你要构建的不是“围绕具体系统自动生成 DoDAF 视图的工具”，而是“一套以参考模型为核心、以 SSOT 为基础、以评审驱动收敛的系统工程知识生产线”。**

---

## 十三、建议的下一步

如果继续推进，最合理的下一步不是继续扩张 Agent 数量，而是先把以下四件事固定下来：

1. **主产物类型清单**：哪些是主产物，哪些是派生产物；
2. **SSOT 最小字段集**：词表、关系、规则、决策各需要哪些字段；
3. **状态机草案**：每类产物从草案到基线的路径；
4. **评审门定义**：哪些问题必须人工裁决，哪些问题可由 Claude Code 自动修补。

完成这一步后，整个系统就会从“研究想法”进入“可持续工程化”的轨道。
