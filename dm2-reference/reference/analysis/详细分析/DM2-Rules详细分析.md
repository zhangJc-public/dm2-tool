## 一、Rules 概述

### 1.1 核心定位

Rules（规则）是 DM2 元模型中描述 **"约束、指导和规范"** 的核心数据组。它定义了活动、执行者和系统的行为边界，是企业架构中保证一致性和合规性的关键机制。

### 1.2 在元模型中的位置

```
┌──────────────────────────────────────────────────────────────┐
│                    DM2 元模型核心结构                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐      │
│   │  Performer  │◄──►│  Activity   │◄──►│    Rules    │      │
│   │  执行者     │    │   活动      │    │   规则      │      │
│   └─────────────┘    └──────┬──────┘    └──────┬──────┘      │
│                             │                    │             │
│         ┌───────────────────┼────────────────────┘           │
│         │  performs         │ ruleConstrains                  │
│         ▼                    ▼                                │
│   ┌─────────────┐    ┌─────────────┐                         │
│   │ Capability  │    │   System    │                         │
│   │   能力      │    │    系统     │                         │
│   └─────────────┘    └─────────────┘                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 Rules 的本质

> **核心原则**：规则是"治理"和"约束"的基础，而非简单的"指令"。

Rules 在 DM2 中体现了**强制性**与**指导性**的层次：
- **强制性规则**：Rule（规则）、Regulation（法规）、Constraint（约束）
- **指导性规则**：Guidance（指南）、Standard（标准）、Agreement（协议）

## 二、核心概念定义

### 2.1 Rule（规则）

> **官方定义**（BMM Concept Catalog / DoDAF）：
> *"A principle or condition that governs behavior; a prescribed guide for conduct or action."*

**关键解读**：
- 规范行为的原则或条件
- 对行动有约束力的指南
- 可被自动化或在人员层面执行

**多来源定义对照**：

| 来源 | 定义 |
|------|------|
| BMM | 业务规则提供具体的、可操作的指导来实现业务策略 |
| dictionary.com | 规范行为、行动、程序、安排的原则或规定 |
| wordnet | 规范行为的原则或条件 |
| wikipedia | 描述组织实现其目标所适用的操作、定义和约束 |

### 2.2 Guidance（指南）

> **官方定义**（DoDAF/CADM / Zachman）：
> *"An authoritative statement intended to lead or steer the execution of actions."*

**关键解读**：
- 权威性的陈述
- 旨在引导或 steering 行动的执行
- 提供方向性的监督

| 来源 | 定义 |
|------|------|
| DoDAF/CADM | 从上级收到的方向性陈述 |
| Zachman | 描述业务流程整体监督的指令类型 |
| Webster | 引导的行为或功能；领导力；方向 |

### 2.3 Standard（标准）

> **官方定义**：
> *"A formal agreement documenting generally accepted specifications or criteria for products, processes, procedures, policies, systems, and/or personnel."*

**关键解读**：
- 正式的文件化协议
- 记录被广泛接受的规格或标准
- 可适用于产品、流程、程序、政策、系统、人员

| 来源 | 定义 |
|------|------|
| DoDAF/CADM | 信息技术标准：用于获取、存储、操作、管理、移动、控制、交换、传输或接收数据的程序、产品或关系的协议 |
| NAF | 经过审核和同行评审的规范，用于指导或约束架构 |
| FEA | 一组标准（部分是强制性的）、自愿性指南和最佳实践 |

### 2.4 Agreement（协议）

> **官方定义**：
> *"A consent among parties regarding the terms and conditions of activities that said parties participate in."*

**关键解读**：
- 各方之间的同意
- 关于参与活动的条款和条件
- 具有约束力的安排

## 三、规则类型层次结构

### 3.1 完整层次结构

根据 DM2 的类型层次，Rules 的完整结构如下：

```
<<IndividualType>>
    GuidanceType（元类型）
         │
         ├── <<IndividualType>> Rule（强制性规则）
         │    │
         │    ├── Policy（策略）
         │    │    │
         │    │    ├── SecurityPolicy（安全策略）
         │    │    ├── AccessPolicy（访问策略）
         │    │    └── UsagePolicy（使用策略）
         │    │
         │    ├── Constraint（约束）
         │    │    │
         │    │    ├── IntegrityConstraint（完整性约束）
         │    │    ├── SecurityConstraint（安全约束）
         │    │    └── TemporalConstraint（时间约束）
         │    │
         │    └── Regulation（法规）
         │         │
         │         ├── Law（法律）
         │         ├── Directive（指令）
         │         └── Mandate（强制性命令）
         │
         ├── <<IndividualType>> Standard（标准）
         │    │
         │    ├── TechnicalStandard（技术标准）
         │    │    ├── DataStandard（数据标准）
         │    │    ├── InterfaceStandard（接口标准）
         │    │    └── SecurityStandard（安全标准）
         │    │
         │    ├── ProcessStandard（过程标准）
         │    │    ├── QualityStandard（质量标准）
         │    │    └── ComplianceStandard（合规标准）
         │    │
         │    └── PersonnelStandard（人员标准）
         │
         ├── <<IndividualType>> Agreement（协议）
         │    │
         │    ├── ServiceAgreement（服务协议）
         │    ├── DataSharingAgreement（数据共享协议）
         │    └── InteroperabilityAgreement（互操作协议）
         │
         └── <<IndividualType>> Guidance（指导性指南）
              │
              ├── BestPractice（最佳实践）
              ├── Guideline（指南）
              └── Procedure（程序）
```

### 3.2 强制性与指导性对比

| 类别 | 类型 | 约束力 | 示例 |
|------|------|--------|------|
| **强制性** | Rule | 必须遵守 | "所有机密数据必须加密存储" |
| | Regulation | 法律强制 | "必须符合 GDPR 数据保护条例" |
| | Constraint | 技术强制 | "密码长度至少8位" |
| | Policy | 组织强制 | "员工必须每年完成安全培训" |
| **指导性** | Standard | 广泛采用 | "推荐使用 AES-256 加密算法" |
| | Agreement | 双方约定 | "服务可用性不低于 99.9%" |
| | Guidance | 方向引导 | "建议采用零信任架构" |

## 四、核心关系

### 4.1 关系一览

| 关系名称 | 关系类型 | 说明 | 方向 |
|---------|---------|------|------|
| `ruleConstrainsActivity` | Overlap | 规则约束活动 | Rule → Activity |
| `guidanceGuidesActivity` | Overlap | 指南指导活动 | Guidance → Activity |
| `standardConstrainsSystem` | Overlap | 标准约束系统 | Standard → System |
| `policyGovernsOrganization` | Overlap | 策略治理组织 | Policy → Organization |
| `agreementConstrainsActivity` | Overlap | 协议约束活动 | Agreement → Activity |
| `rulePartOfMeasureType` | WholePart | 规则是度量类型的组成部分 | Rule → MeasureType |

### 4.2 关系详解

#### 4.2.1 ruleConstrainsActivity

> **定义**：An overlap between a Rule and the Activities it allows.

**语义解读**：
- Rule 与 Activity 之间存在重叠关系
- 表示该规则"允许"或"约束"某些活动
- 是规则应用于活动的核心机制

```
┌─────────────────────────────────────────────────────────────┐
│                   ruleConstrainsActivity                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────┐                                         │
│   │    Rule     │                                         │
│   │   规则      │                                         │
│   └──────┬──────┘                                         │
│          │ overlap (constrains)                            │
│          ▼                                                 │
│   ┌─────────────┐                                         │
│   │  Activity   │                                         │
│   │   活动      │                                         │
│   └─────────────┘                                         │
│                                                              │
│   例：安全规则约束情报收集活动的执行条件                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 4.2.2 guidanceGuidesActivity

> **定义**：An authoritative statement intended to lead or steer the execution of actions.

**语义解读**：
- Guidance 与 Activity 之间存在引导关系
- 提供行动的方向性指导
- 不具有强制性，但具有权威性

#### 4.2.3 standardConstrainsSystem

> **定义**：Standard 作为系统设计的约束条件。

**语义解读**：
- Standard 约束系统的设计和实现
- 包括技术标准、数据标准、安全标准等
- 是系统合规性评估的基础

#### 4.2.4 policyGovernsOrganization

> **定义**：Policy 治理组织的运作。

**语义解读**：
- Policy 定义组织的治理框架
- 约束组织结构和行为
- 通常具有长期性和稳定性

#### 4.2.5 agreementConstrainsActivity

> **定义**：Agreement 作为活动参与的条款和条件。

**语义解读**：
- 协议定义活动参与的条件
- 约束多个参与方之间的行为
- 体现各方的一致性同意

## 五、规则与活动的关系模型

### 5.1 规则约束活动的三种模式

```
┌─────────────────────────────────────────────────────────────┐
│              规则约束活动的三种模式                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  模式 1：禁止（Prohibition）                                 │
│  ┌─────────────┐                 ┌─────────────┐           │
│  │    Rule     │ ── prohibits ──►│  Activity   │           │
│  │  "禁止X行为" │                 │    活动     │           │
│  └─────────────┘                 └─────────────┘           │
│                                                              │
│  模式 2：要求（Requirement）                                 │
│  ┌─────────────┐                 ┌─────────────┐           │
│  │    Rule     │ ── requires ───►│  Activity   │           │
│  │  "必须Y操作" │                 │    活动     │           │
│  └─────────────┘                 └─────────────┘           │
│                                                              │
│  模式 3：许可（Permission）                                  │
│  ┌─────────────┐                 ┌─────────────┐           │
│  │    Rule     │ ── permits ────►│  Activity   │           │
│  │  "允许Z行为" │                 │    活动     │           │
│  └─────────────┘                 └─────────────┘           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 规则应用于活动的判定逻辑

```
活动可执行 当且仅当:
┌─────────────────────────────────────────────────────────────┐
│  1. 没有禁止规则覆盖该活动                                    │
│     AND                                                    │
│  2. 所有要求规则都已满足                                      │
│     AND                                                    │
│  3. 存在至少一个许可规则覆盖该活动                             │
└─────────────────────────────────────────────────────────────┘
```

## 八、规则建模实践

### 8.1 StdV-1 标准配置视图

在 DoDAF StdV-1（标准配置）中，Rules 建模需要描述：

| 要素 | 说明 | 必选/可选 |
|------|------|----------|
| 标准标识 | 标准的唯一标识符 | 必选 |
| 标准类型 | Regulation/Policy/Standard/Rule/Agreement | 必选 |
| 适用范围 | 约束的活动/系统/组织 | 必选 |
| 约束关系 | constrains/guides/governs | 必选 |
| 版本信息 | 标准的版本和日期 | 可选 |
| 合规要求 | 符合该标准的要求 | 可选 |

### 8.2 规则建模步骤

```
步骤 1：识别约束源
   └─► 确定法规、政策、标准、协议等约束来源

步骤 2：分类规则
   └─► 归类到适当的规则类型（Regulation/Policy/Standard/Rule/Agreement）

步骤 3：建立约束关系
   └─► 用 constrains/governs/guides 连接规则与被约束对象

步骤 4：定义规则细节
   └─► 规则的具体内容、参数、条件

步骤 5：验证一致性
   └─► 检查规则之间的冲突和冗余

步骤 6：文档化
   └─► 生成 StdV-1 标准配置视图
```

## 十、与 DoDAF 1.5 的差异

### 10.1 规则建模的演进

| 方面 | DoDAF 1.5 | DoDAF 2.0 (DM2) |
|------|----------|-----------------|
| **规则定义** | 模糊，主要关注信息交换 | 明确的类型层次结构 |
| **约束机制** | 隐含在活动描述中 | 显式的 constrains 关系 |
| **标准组织** | 分散 | 统一的 StdV-1 标准配置视图 |
| **层次结构** | 扁平 | 层次化（法规→策略→标准→规则→协议） |

### 10.2 关键改进

1. **类型细分**：区分 Rule、Guidance、Standard、Agreement、Policy、Regulation
2. **显式约束关系**：通过 constrains/governs/guides 显式建模
3. **可追溯性**：从法规到具体规则的可追溯链条
4. **视图整合**：StdV-1 作为规则的标准配置视图

## 十一、关键洞察总结

### 11.1 核心要点

| # | 要点 | 说明 |
|---|------|------|
| 1 | **规则是约束机制** | Rules 通过 constrains/governs/guides 约束活动、系统、组织 |
| 2 | **类型层次完整** | Regulation → Policy → Standard → Rule → Agreement |
| 3 | **强制与指导分离** | Rule/Regulation/Constraint 强制性 vs Standard/Agreement/Guidance 指导性 |
| 4 | **规则是元模型公民** | Rules 约束活动、资源、执行者、服务等所有实体 |
| 5 | **可追溯链** | 从高层法规到具体业务规则的可追溯链条 |

### 11.2 建模检查清单

```
□ 识别所有适用的法规（Regulation）
□ 定义组织级策略（Policy）
□ 选择适用的标准（Standard）
□ 制定具体的业务规则（Rule）
□ 建立各方协议（Agreement）
□ 建立约束关系（constrains/governs/guides）
□ 验证规则的一致性和完整性
□ 文档化到 StdV-1 视图
□ 确保规则的可执行性和可审计性
```
