# SV-10a: Systems Rules Model（系统规则模型）

## 一、定位与目的

### 1.1 这个模型回答什么问题？

**一句话**：系统行为受哪些技术性规则约束？

SV-10a 是系统视点的**规则库**——捕获约束系统行为的**技术性和实现级规则**。

核心问题集：
| 问题 | 回答 |
|------|------|
| 系统在什么条件下必须/禁止做什么？ | 技术约束规则 |
| 这些规则的权威依据是什么？ | 标准/规范引用 |
| 系统行为是否满足安全和合规要求？ | 合规检查基础 |

### 1.2 与 OV-6a 的分工 ⭐⭐

这是理解 SV-10a 最关键的概念：

| 维度 | **OV-6a** (作战规则) | **SV-10a** (系统规则) |
|------|---------------------|----------------------|
| **关注点** | **What**（做什么约束） | **How**（如何实施约束） |
| **规则层级** | 条令/业务/使命级 | **技术/实现级** |
| **示例** | "BDA 仅在晴天执行" | "加密强度 ≥256 位；TLS 1.3 only" |
| **来源** | 条令/指南/交战规则 | 技术标准/接口规范/安全策略 |
| **抽象程度** | 高（人类可读） | 低（机器可执行） |

> **OV-6a 是 What，SV-10a 是 How。**

### 1.3 规则类型示例

| 类别 | 示例规则 |
|------|---------|
| **接口规则** | "所有 API 必须通过 TLS 1.3 通信" |
| **数据规则** | "PII 字段必须 AES-256 加密存储" |
| **访问控制** | "管理员操作必须双因素认证" |
| **性能规则** | "响应时间不得超过 200ms (P99)" |
| **兼容性规则** | "必须支持 JSON 和 XML 双格式" |

## 二、核心内容解析

### 2.1 规则表达

与 OV-6a 类似，SV-10a 规则也可采用两种形式：

| 形式 | 结构 | 示例 |
|------|------|------|
| **命令性** | "必须……" | "所有外部接口**必须**通过 API Gateway" |
| **条件命令性** | "如果……则……" | "如果数据包含 PII，**则必须**加密后传输" |

但 SV-10a 的规则通常更加**形式化**，以便：
- 自动化合规检查
- 代码生成
- 测试用例推导

### 2.2 与 StdV-1 的关系

SV-10a 的规则往往**引用** StdV-1 中的标准作为权威来源：
- StdV-1: "本架构采用 FIPS 140-2"
- SV-10a: "所有加密模块**应符合** FIPS 140-2 Level 3 要求"

### 2.3 与 SvcV-10a 的对称

SV-10a（系统规则）和 SvcV-10a（服务规则）是同一抽象级别的**孪生视图**：
- SV-10a 聚焦**系统**层面的技术约束
- SvcV-10a 聚焦**服务**层面的技术约束
- 在 SOA 架构中两者可能有大量重叠

## 三、关联视图

| 上游依赖 | 下游支撑 | 同级互补 |
|----------|---------|---------|
| StdV-1（标准来源） | → 合规验证 | OV-6a（What vs How） |
| SV-1/SV-4（约束对象） | → 安全测试用例 | SvcV-10a（系统 vs 服务） |
| SV-6（安全属性） | → | |

### 规则体系的完整分层

```
战略/条令层:
  OV-6a (作战规则) = "What: 使命约束"

        ↓ 技术转化

实现层:
  SV-10a (系统规则)  = "How: 系统如何遵守"
  SvcV-10a (服务规则) = "How: 服务如何遵守"

        ↓ 代码实现

运行时代码:
  配置文件 / 策略引擎 / 访问控制列表
```
