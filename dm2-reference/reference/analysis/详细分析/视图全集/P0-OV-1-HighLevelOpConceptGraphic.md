# OV-1: High Level Operational Concept Graphic（高层作战概念图）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这个架构描述的是**什么场景**？涉及哪些参与者和操作？

OV-1 是作战视点的**图形化执行摘要**——一张图说清"这是什么、谁参与、怎么运作"。

核心问题集：
| 问题 | 回答 |
|------|------|
| 这是什么任务/场景？ | 任务或使命类别的图形化描述 |
| 涉及哪些关键角色？ | 参与者和操作方 |
| 与外部环境/系统如何交互？ | 架构边界和接口关系 |
| 有什么独特之处？ | 关键或有趣的作战概念亮点 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────┐
│          OV-1: 高层作战概念图                  │
│     （图形化执行摘要 / 人际沟通工具）           │
├──────────────────────────────────────────────┤
│                                              │
│   ┌──────────────────────────────────┐       │
│   │         [场景图形]               │       │
│   │  (任务/参与者/环境/外部系统)      │       │
│   └──────────────────────────────────┘       │
│                    ↑                          │
│            AV-1 的文字内容                    │
│        描述 OV-1 图中的概念                  │
│                                              │
│   特点:                                      │
│   ├─ 最通用、最灵活格式的架构模型             │
│   ├─ 主要用途是辅助人类沟通                  │
│   ├─ 面向高层决策者展示                      │
│   └─ 图形 + 配套文字说明（缺一不可）         │
│                                              │
└──────────────────────────────────────────────┘
```

**与 AV-1 的特殊关系 —— 1:N 模型**：

> ⭐ **核心发现：一个架构描述中，AV-1 只有 1 份，但 OV-1 可以有 N 份。**

OV-1 是 **AV-1 文字内容的图形化表示**。两者互补：
- **AV-1** = 结构化的文字描述（详细信息密度高）—— **整个项目只有 1 份**
- **OV-1** = 一张图传达整体概览（视觉冲击力强）—— **可以有多个，按场景分解**

⚠️ **仅靠图形不足以捕获所有必要的架构数据**——配套文字说明至关重要。

#### 1.2.1 为什么需要多个 OV-1？

DoDAF 规范原文定义：

> "The OV-1 describes **a** mission, class of mission, or **scenario**."
> — DoDAF v2.02 Vol.II, p.142

关键词是 **"a"**（单数）—— 一个 OV-1 聚焦于**一个特定场景或使命域**。

当一个架构描述涵盖多个作战场景、多个任务域、或多个阶段时，单一图形无法清晰表达全部内容，此时就需要将 OV-1 分解为**多个独立的概念图**，每张聚焦一个子场景。

#### 1.2.2 权威实践佐证

| 来源 | 论述 |
|------|------|
| **知乎·DoDAF实践** | > "一个项目中只有**一张 AV-1** 视图；OV-1 通常情况下则**有多张**。例如：为论证一艘新型护卫舰，用 **1 张 AV-1** 说明经费/技术/目的等背景信息；用 **3 张 OV-1** 分别描述护卫舰的**反潜、防空、反舰**作战构想。" |
| **UML.org.cn 教程** | > "OV-1 描述**一个**任务、任务类别或场景……是 AV-1 概述和汇总信息的书面内容的图形表示。" |
| **DoDAF v2.02 PDF p.142** | OV-1 是"**最通用的架构模型，格式上最灵活的**"——灵活性本身就暗示了可以按需创建多份 |

#### 1.2.3 AV-1:OV-1 = 1:N 关系图

```
┌──────────────────────────────────────┐
│           AV-1 (唯一)                │
│    架构概述 / 背景 / 范围 / 目的     │

           ├───────────────────────────┐

#### 1.2.4 多 OV-1 的分解维度

当决定是否创建多个 OV-1 时，可按以下维度拆分：

| 分解维度 | 示例 | 判断标准 |
|---------|------|---------|
| **按任务域/能力域** | 反潜 / 防空 / 反舰 / 对陆打击 | 各域的参与者和流程差异显著 |
| **按作战阶段** | 威慑阶段 / 交战阶段 / 战后恢复 | 不同阶段的关注点和时间线完全不同 |
| **按地理位置/战区** | 印太战区 / 欧洲战区 / 中东战区 | 各战区的环境和外部系统不同 |
| **按基线/目标/过渡** | As-Is 概念 / To-Be 概念 / 过渡路径 | 架构转型叙事需要对比展示 |
| **按受众层次** | 决策者版(高度抽象) / 技术管理者版(中等细节) | 不同受众需要不同的信息密度 |

#### 1.2.5 实践指南

✅ **建议创建多 OV-1 的信号**：
- 架构描述覆盖 **≥2 个独立的任务域或作战场景**
- 单一图形的信息密度超过可读阈值（一般 ≤7±2 个核心要素）
- 不同利益相关者群体关注**完全不同的方面**
- 需要分别描述 **As-Is 和 To-Be** 状态

❌ **不建议拆分的情形**：
- 架构主题单一且聚焦（如单一系统的升级改造）
- 强行拆分导致各图之间大量重复内容
- 受众只需要一份概览性材料

💡 **命名建议**：当存在多个 OV-1 时，使用带限定词的命名方式：
- `OV-1a: 高层作战概念图 - 反潜作战场景`
- `OV-1b: 高层作战概念图 - 防空作战场景`
- `OV-1c: 高层作战概念图 - 反舰作战场景`

## 二、核心内容解析

### 2.1 必要性说明

> OV-1 描述一个**任务、任务类别或场景**。它展示主要的作战概念以及操作的有趣或独特方面。它描述**主题架构与其环境之间的交互**，以及**架构与外部系统之间的交互**。
>
> OV-1 提供关于架构是什么的**图形化描绘**，以及对涉及的参与者和操作的概念。

预期用途：
1. 将作战情况或场景置于上下文中
2. 作为讨论和演示的工具（例如促进行业参与采购）
3. 作为已发布架构中更详细信息的**高层汇总插图**

### 2.2 推荐表达形式

| 形式 | 适用场景 | 备注 |
|------|---------|------|
| **概念图/场景图** | ⭐ 默认形式 | 自由格式，最灵活 |
| 场景草图 | 早期阶段 | 快速沟通想法 |
| 信息图 | 面向非技术受众 | 高层决策者友好 |
| 视频/动画 | 复杂动态场景 | 官方明确支持 |

> OV-1 是**最通用的架构模型，格式上最灵活的**。

### 2.3 关键数据元素

| 内容要素 | 说明 |
|---------|------|
| 业务活动或任务 | 核心操作内容 |
| 高层操作 | 操作层级的关键行为 |
| 组织 | 参与的组织实体 |
| 资产的地理分布 | 空间布局 |
| 与环境的交互 | 外部因素影响 |
| 与外部系统的交互 | 跨边界的接口 |

#### 性能参数容器功能

> OV-1 在为相关作战模型建立上下文方面很有用。此上下文可以以阶段、时间段、使命和/或位置的形式存在。特别是，它为**时空约束的性能参数（度量）提供了容器**。

示例：
- 第一阶段的沙漠作战性能参数可能与第二阶段不同
- 第二阶段的丛林作战参数可能与同阶段的沙漠作战不同

### 2.4 完整细节

#### 版本演进策略（与 AV-1 类似）

| 版本时机 | 内容重点 |
|---------|---------|
| **初始版本** | 聚焦范围、图示范围 |
| **中间版本** | 反映架构开发中识别的范围调整和其他变更 |
| **最终版本** | 总结发现，向高层决策者呈现 |
| **特殊情况** | 有时 OV-1 是**最后开发的模型**——因为它汇总整个架构描述的摘要信息 |

#### 业务能力适配

> 当架构描述的主题是**业务能力而非战场空间能力**时，术语使用需要一些调整。然而，拥有高层（业务）作战概念的思路仍然有意义，OV-1 中的图形表示对可能创建的更结构化模型增加价值。

### 3.1 与其他视图的数据流向（含多 OV-1 场景）

```
                    ┌─────────────┐
                    │    CV-1     │ 能力愿景
                    │  (战略方向)  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │  OV-1 #1  │ │  OV-1 #2  │ │  OV-1 #N  │ ← 多个概念图
       │ (场景A)   │ │ (场景B)   │ │ (场景N)   │   各聚焦不同

             └────────────┼────────────┘

              ┌──────────────────────────┐
              │         AV-1             │ ← 唯一的架构概述

              └──────────────────────────┘
