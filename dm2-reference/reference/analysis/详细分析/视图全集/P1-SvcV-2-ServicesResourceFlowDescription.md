# SvcV-2: Services Resource Flow Description（服务资源流描述）

## 一、定位与目的

### 1.1 这个图回答什么问题？

**一句话**：服务之间交换什么资源？通过什么协议和端口连接？

SvcV-2 是服务视点的**接口规格图**——精确描述服务间的连接细节（协议栈、端口、资源流）。

核心问题集：
| 问题 | 回答 |
|------|------|
| 服务 A 的哪个端口连到服务 B？ | 端口级连接规格 |
| 使用什么通信协议？ | 协议栈定义 |
| 交换了哪些资源流？ | 资源流清单 |
| 连接是现有的还是规划的？ | 状态标记 |

### 1.2 与 SV-1/SV-2 的镜像关系 ⭐

| 维度 | **SV-1/SV-2** (系统) | **SvcV-2** (服务) |
|------|---------------------|-------------------|
| **交互模式** | 点对点系统接口 | **发布-订阅** 服务模式 |
| **焦点** | 系统→系统的物理连接 | 服务→服务的逻辑契约 |
| **粒度** | 接口/协议/端口 | 服务端点/协议栈 |
| **适用架构** | 单体/集成架构 | SOA / 微服务 / 云原生 |

## 二、核心内容解析

### 2.1 核心要素

> 对于网络数据服务，SvcV-2 包括**服务、端口和服务之间的资源流**。

每个 SvcV-2 模型可展示：

| 要素 | 说明 |
|------|------|
| **连接的端口** | 哪些端口互相连接 |
| **生产方服务** | 端口所属的服务 |
| **消费方服务** | 资源流被哪个服务消费 |
| **资源流定义** | 物理/逻辑连接性 + 使用的协议 |
| **协议栈** | 连接中使用的完整协议层次 |

### 2.2 非IT服务的支持 ⭐

> SvcV-2 也可用于描述**非 IT 类型**的服务，如**搜索与救援（Search and Rescue）**。

这意味着 DoDAF 的服务概念**不限于软件服务**——任何"为消费者提供价值的能力单元"都可以建模为服务。

### 2.3 表达灵活性

架构师可以选择：

| 方式 | 适用场景 |
|------|---------|
| 每个服务资源流一张图 | 复杂场景，需要详细展示 |
| 按生产+消费配对出图 | 关注特定交互对 |
| 所有流在一张图中 | 简单场景，全局概览 |

### 2.4 网络即服务 ⭐

> **网络被表示为服务**。架构师可以选择将其他服务展示为网络的组成部分（如果它们属于网络基础设施）。

这体现了 DoDAF V2.0 的面向服务思想——基础设施本身也是一种服务。

### 2.5 与 StdV-1 的关联

> SvcV-2 图中引用的任何协议都需要在 **StdV-1 Standards Profile** 中定义。

确保协议引用的可追溯性和权威性。
