# OV-5a/OV-5b: Operational Activity Decomposition Tree & Model（作战活动分解树与活动模型）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：为了完成任务，需要**做什么活动**？这些活动如何分解和关联？

OV-5a/b 是作战视点的**核心过程模型**——定义"what work is required"，独立于"how it is carried out"。

| 模型 | 回答的问题 | 形式 |
|------|-----------|------|
| **OV-5a** | 有哪些活动？如何分层？ | 树形结构（导航用） |
| **OV-5b** | 活动之间如何通过资源流连接？ | 活动图（分析用） |

### 1.2 在架构描述中的角色

```

├──────────────────────────────────────────────────┤
│                                                  │
│   OV-5a (分解树):              OV-5b (活动模型):   │
│                                                  │
```

**关键角色**：
- **业务过程的完整描述**

- **OV-6 系列的基础输入**（活动排序/规则/追踪的前提）
- **能力映射的目标端**（CV-6 映射到这些活动）

## 二、核心内容解析

### 2.1 必要性说明

> OV-5a 和 OV-5b 描述在实现使命或业务目标过程中**通常进行的操作**。它描述**作战活动（或任务）**；活动之间的输入/输出资源流；以及与架构范围外活动的交换。
>
> ⭐ **核心理念**：Operational Activity 是"**需要做什么工作**"，**独立于如何执行**来规定。为保持这种独立性，使用 OV-2 中的逻辑活动和位置来表示执行 Operational Activities 的结构。
>
> **Operational Activities 被实现（realized）为**：
> - **System Functions**（SV-4 Systems Functionality Description）—— 即 "how"
> - **Service Functions**（SvcV-4 Services Functionality Description）—— 以执行它们的资源来规定

预期用途：
1. 活动和工作流描述
2. 需求捕获
3. 角色和职责定义
4. 支持任务分析以确定培训需求
5. 问题空间定义
6. 作战规划
7. 后勤保障分析
8. 信息流分析

#### 额外的预期用途（官方列表）：

- 结合 OV-2 时**清晰划定活动责任线**
- **发现不必要的作战活动冗余**
- 做出关于精简、合并或省略活动的决策
- 定义或标记需要进一步审查的问题、机会或作战活动及其交互

### 2.2 推荐表达形式

官方明确 **不 endorsement 特定方法论**：

| 方法 | 适用场景 | 说明 |
|------|---------|------|
| **IDEF0** | 功能建模经典方法 | 官方提及的首选之一 |
| **UML Class Diagrams** | 面向对象风格 | 活动作为类 |
| **树形图** | 仅 OV-5a | 分层导航 |
| **活动图** | OV-5b | 资源流连接 |

#### OV-5a vs OV-5b 的两种基本描绘方式：

| 类型 | 内容 | 用途 |
|------|------|------|
| **Activity Decomposition Tree** | 树形结构的活动 | 提供**导航辅助** |
| **Activity Model** | 通过 Resource Flows 连接的活动 | 支持 OV-3 开发 |

> OV-5a 帮助提供涉及活动的整体图景和快速导航 OV-5b 的参考。

### 2.3 关键数据元素

| 元素 | 说明 | 来源视图映射 |
|------|------|-------------|
| **Operational Activity** | 独立于实现的活动定义 | → SV-4 SysFunction / SvcV-4 SvcFunction |
| **Input/Output flows** | 活动间的资源交换 | → DIV-1/DIV-2 信息实体; OV-3 详细记录 |
| **External Interchanges** | 与范围外活动的 I/O | → 架构边界定义 |

### 2.4 完整细节

#### 2.4.1 Standard Operational Activities（标准作战活动）

> OV-5a 或 OV-5b 中描述的活动是**标准作战活动**——它们被映射到 CV-6 Capability to Operational Activities Mapping 中对应的能力。
>
> **标准作战活动**是在条令中定义但**不为特定系统定制**的活动——即它们足够通用，可以在不排除一系列可能解决方案的情况下使用。

这意味着 OV-5 中的活动应该是：
- ✅ 条令化的、通用的
- ✅ 不绑定到特定系统实现
- ❌ 不是某个具体软件的功能规格

#### 2.4.2 OV-5 与 OV-2 的互补开发

> OV-5 和 OV-2 在一定程度上是**互为补充**的。OV-5 关注**作战活动**，而 OV-2 关注**与位置相关的作战活动**。由于位置和作战活动之间的关系，这两类模型通常应**共同开发**。

#### 2.4.3 业务建模适用性

> OV-5a 和 OV-5b 同样适用于描述**非军事活动**，预计将被广泛用于**业务建模**。

## 三、关联视图

| 上游依赖（输入） | 下游支撑（输出） | 同级互补 |
|------------------|-----------------|---------|
| CV-6（能力→活动映射的目标端） | → SV-4（系统功能的逻辑来源） | OV-2（资源流上下文） |
| OV-2（位置+活动联合开发） | → SvcV-4（服务功能的逻辑来源） | OV-4（角色→活动分配） |
| OV-4（组织/角色） | → OV-3（活动间资源流的来源） | OV-1（场景上下文） |
| 条令文件（标准活动来源） | → OV-6a/b/c（活动序列/规则/追踪基础） | |

### 活动模型的下游链

```
OV-5 (作战活动) ──实现(Realize)──→ SV-4 (系统功能)
                              ──实现(Realize)──→ SvcV-4 (服务功能)

OV-5 活动间资源流 ──详细化──→ OV-3 (资源流矩阵)
                        ──语义化──→ DIV-1/2 (信息/数据模型)
```
