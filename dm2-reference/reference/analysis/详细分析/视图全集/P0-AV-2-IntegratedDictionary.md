# AV-2: Integrated Dictionary（集成词典）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这份架构里用到的**所有术语是什么意思？**

AV-2 是整个架构描述的**术语基础设施**——一个集中的元数据仓库，确保所有视点、模型和视图中的数据元素定义一致、来源可追溯。

核心问题集：
| 问题 | 回答 |
|------|------|
| "XX"在这个架构中指什么？ | 文本定义 + 层级位置 |
| 这个术语来自哪里？ | 权威来源引用 |
| 不同视图里的"XX"是同一个东西吗？ | 唯一标识符交叉引用 |
| 有没有引入新术语？ | 非标准术语标记 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────────┐
│                 AV-2: 集成词典                    │
│            （术语权威源 / 一致性基础）              │
├──────────────────────────────────────────────────┤
│                                                  │
│   ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐           │
│   │ CV-2 │ │ OV-5 │ │ SV-4 │ │ SvcV │  ...     │
│   │能力  │ │活动  │ │功能  │ │服务  │           │
│   └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘           │
│      │       │       │       │                   │
│      └───────┴───────┴───────┘                   │
│                  │                               │
│           所有术语 → AV-2 统一注册                │
│                                                  │
└──────────────────────────────────────────────────┘
```

**关键角色**：
- **跨视图一致性保障器**——防止同名异义或同义异名
- **联邦互操作基础**——多个架构描述联合时，术语一致性是前提
- **开发加速器**——标准分类法作为构建模块复用
- **验证与审计工具**——追溯每个元素到权威数据源

### 为什么 AV-2 至关重要？

官方明确指出：**早期 DoD 架构开发的经验教训**表明，不同术语表示相同架构数据使得**架构联邦化（Federation）极为困难**。

## 二、核心内容解析

### 2.1 必要性说明（官方 Purpose 翻译）

> AV-2 展示了架构中使用的**所有元数据**。它以层级方式呈现所有数据，为每个元素提供文本定义，并引用其来源（如 DoDAF 元模型、IDEAS、发布文档或政策）。
>
> AV-2 展示了在架构描述中被实例化的 DM2 概念，以及由架构描述**新引入的元素**（即不在 DM2 中的概念）。

使用标准分类法相比自由文本标签的**优势**：
1. 在基于 DoDAF 模型的填充视图内部保持一致
2. 在不同的架构描述之间保持一致
3. 促进架构描述的开发、验证、维护和复用
4. 将架构数据追溯到权威数据源

### 2.2 推荐表达形式

| 形式 | 适用场景 | 备注 |
|------|---------|------|
| **层级结构/树形** | ⭐ 默认形式 | 展示术语间的 sub-supertype 关系 |
| 表格式附录 | 属性较多时 | 能力属性/度量等详细数据 |
| 数据库/电子表格 | 大型架构 | 支持排序、筛选、查询 |

### 2.3 关键数据元素（DM2 映射）

AV-2 是**整个 DM2 的投影面**。官方建议按以下 DM2 区域组织分类法：

#### 必需的分类法领域（Taxonomy Areas）

| # | 分类法领域 | 最小内容要求 | 关联 DM2 数据组 |
|---|----------|------------|----------------|
| 1 | **Capabilities（能力）** | 名称、描述、性能度量条件 | DM2-Capability |
| 2 | **Resource Flow（资源流）** | 名称、信息元素描述、分解为组成部分和子类型、映射到系统数据交换元素 | DM2-ResourceFlow |
| 3 | **Activities - Operational（作战活动/任务）** | 名称、描述、分解为组成部分 | DM2-Activity（Operational） |
| 4 | **Activities - System/Service Functions（系统/服务功能）** | 名称、描述、分解为组成部分 | DM2-Activity（System） |
| 5 | **Performance Parameters（性能参数）** | 名称、描述、度量单位、适用条件 | DM2-Measure |
| 6 | **Performers（执行者）** | 名称、描述、分解（如服务包含其他服务）、分类 | DM2-Performer |
| 7 | **Skills（技能）** | 名称、描述、度量单位 | DM2-Measure |
| 8 | **Standards（标准）** | 标准类别（如 DISR 服务域） | DM2-Rules |
| 9 | **Triggers/Events（触发器/事件）** | 名称、描述、分解、类型分类 | DM2-Activity（事件相关） |

### 2.4 完整细节

#### 2.4.1 分类法的构建原则

**与 AV-2 的关键区别**：
- **CV-2** 只使用一种特化关系：**sub-supertype**（纯特化）
- **AV-2** 可以包含更丰富的层级结构和元数据

**时间维度**：
> DoD 内的组织、服务、系统和活动持续演化。使用已建立、经验证的分类法结构的价值在于可以按需扩展或收缩。随着对分类法理解的加深，**新模型的开发将大大简化**。

**标准分类法的"建筑块"作用**：
> 标准分类法（如 DISR Service Categories）成为更全面、高质量的 DoDAF 模型和 Fit-for-Purpose 视图的建筑块。

**推荐的外部权威分类法来源**：
- FEA Reference Models（联邦企业架构参考模型）
- Joint Common System Function List（联合通用系统功能清单）
- DoD Extensible Markup Language Registry and Clearinghouse
- Net-Centric Implementation Document (NCID)

#### 2.4.2 本地词汇的处理策略

⚠️ **重要警告**：

> 特定社区可能有自己的**本地操作词汇**。这些本地词汇可能与其他操作社区**以完全不同的方式使用相同的术语**。
>
> 示例："track"一词在**航母战斗群社区**和**扫雷艇社区**中指代截然不同的概念——但两者都是海军作战群体，可能一起参与近岸作战任务。

**处理原则**：
1. 架构描述内部的模型和视图应使用**本地操作社区的词汇**以获得社区认可
2. 数据元素必须**唯一识别并在所有视点/模型/视图中一致使用**
3. 填充视图应对任何独特定义添加注释
4. 尽可能提供到标准定义的**映射关系**

## 三、关联视图

| 上游依赖（输入） | 下游支撑（输出） | 同级互补 |
|------------------|-----------------|---------|
| 所有视图的术语需求 | ← 所有视图（术语权威源） | AV-1（概览） |
| 外部权威分类法 | → 架构联邦化的基础 | |

### AV-2 与其他视图的关系模式

```
CV-2 (能力分类法) ──→ AV-2 的 Capabilities 部分
OV-5a/b (活动模型) ──→ AV-2 的 Activities-Operational 部分
SV-4 (系统功能)    ──→ AV-2 的 Activities-System 部分
SvcV-4 (服务功能)  ──→ AV-2 的 Activities-Service 部分
StdV-1 (标准概况)  ──→ AV-2 的 Standards 部分
OV-2/SV-2 (资源流) ──→ AV-2 的 Resource Flow 部分
所有含 Performer 的视图 → AV-2 的 Performers 部分
```
