# OV-3: Operational Resource Flow Matrix（作战资源流矩阵）

## 一、定位与目的

### 1.1 这个矩阵回答什么问题？

**一句话**：作战活动之间具体交换了什么资源？属性是什么？

OV-3 是 OV-2（资源流描述）的**详细化版本**——从"谁需要和谁交换"深入到"交换什么、为何需要、关键属性是什么"。

核心问题集：
| 问题 | 回答 |
|------|------|
| 活动间交换了哪些具体资源？ | 资源流清单 + 属性 |
| 这些资源的时延/密级/关键度？ | 关键属性追踪 |
| 一条 Needline 对应多少条资源流？ | N:1 映射关系 |
| 跨能力边界的资源流有哪些？ | 重点关注的互操作需求 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────┐
│       OV-3: 作战资源流矩阵                     │
│    （OV-2 的详细化 / 属性规格表）             │
├──────────────────────────────────────────────┤
│                                              │
│   OV-2 (图形):              OV-3 (表格):      │
│   ┌───┐ ──Needline──→ ┌───┐    ┌──────────┬──────┬──────┬──────┐ │
│   │ A │                │ B │    │ 源活动  │目标  │资源  │属性  │ │
```

## 二、核心内容解析

### 2.1 必要性说明

> OV-3 处理**作战活动和位置之间交换的作战资源流**。
>
> 资源流提供了与所关注作战能力的互操作需求的**进一步细节**。
>
> **焦点是跨能力边界的资源流**。

核心价值链：
```
OV-2 (Needline 网络) 
    ↓ 提取并细化
OV-3 (资源流属性矩阵)
    ↓ 输入到
SV-2/SvcV-2 (系统/服务资源流的实现)
```

### 2.2 OV-3 vs OV-2 的关系 ⭐⭐

这是理解 OV-3 最关键的点：

| 维度 | OV-2 | OV-3 |
|------|------|------|
| **形式** | 图形（节点+连线） | 表格（行列+属性） |
| **粒度** | Needline 级别（逻辑需求） | 资源元素级别（具体内容） |
| **回答** | "谁和谁之间有交换需求？" | "交换了什么？属性如何？" |
| **映射** | 一条 Needline | 可能对应**多条**资源流 |
| **焦点** | 协作关系的拓扑 | 资源内容的属性 |

> **OV-3 不是 OV-2 的替代品，而是其详细化补充。**

#### N:1 映射关系

规范明确指出：

> **OV-3 Resource Flows 到 OV-2 Needlines 不总是一对一映射；相反，多条独立的 Resource Flows 可能关联到一条 Needline。**

示例：一条"A→B"的 Needline 下可能包含：
- 目标数据流（时效 <30s）
- 态势报告流（时效 <5min）
- 控制指令流（时效 <10s）
- 状态确认流（时效 <1min）

这 4 条资源流在 OV-3 中各自独立记录属性，但在 OV-2 中只显示为一条连线。

### 2.3 核心内容要素

OV-3 通过以下维度描述每次资源交换：

| 要素 | 说明 |
|------|------|
| **源活动** | 生产资源的作战活动 |
| **目标活动** | 消费资源的作战活动 |
| **位置** | 活动/资源所在的地理位置 |
| **资源元素** | 具体交换的内容（见 DIV-2） |
| **交换原因** | 为何该资源是必需的 |
| **关键属性** | 时效、密级、关键度等 |

#### 资源元素的来源

> 资源元素（Resource Element）是受作战流程约束的资源流的**形式化表示**（定义见 DIV-2 Logical Data Model）。

资源元素可以：
- 中介活动和依赖流（参见 OV-5b）
- 由表达协作关系的 Needline 承载
- 同一资源元素可在多条 Resource Flow 中复用

### 2.4 关键属性的选取原则 ⭐

> 对作战任务至关重要的资源流方面将作为**属性**在 OV-3 中追踪。

属性选取取决于**架构主题**：

| 架构主题 | 关键属性示例 |
|---------|------------|
| **战术战场目标** | 敌方目标信息的**及时性（Timeliness）** |
| **安全架构** | **关键性（Criticality）** 和 **密级（Classification）** |
| **后勤保障** | 数量、体积、重量 |
| **指挥控制** | 完整性、优先级、访问权限 |

### 2.5 安全架构的特殊要求 ⚠️⚠️

规范特别强调：

> 在**安全架构**的背景下，重要的是**识别每一种可能的且必需的交换**。

这意味着：
- ❌ 普通场景：只关注最重要的资源流
- ✅ 安全场景：**穷举所有交换**（不能遗漏任何一条）

### 2.6 范围界定

> OV-3 **不意图**成为与架构描述相关的每个作战活动和位置的每条资源流的所有细节的详尽列表。
>
> 相反，本模型旨在捕获所选资源流的**最重要方面**。

这是一个重要的**实用性原则**——不要试图穷举一切（除非是安全架构），聚焦于关键资源流。

### 2.7 表达形式

> OV-3 信息可以以**表格形式呈现**。DoDAF V2.0 **不规定** OV-3 矩阵的列标题。

这意味着列头可以根据项目需求自定义，常见的包括：

| 常见列 | 可选列 |
|--------|--------|
| 源活动/位置 | 数据格式 |
| 目标活动/位置 | 交换频率 |
| 资源名称 | 传输协议 |
| 资源类型 | QoS 要求 |
| 交换原因 | 安全标记 |
| 关键属性 | 备选方案 |

## 三、关联视图

| 上游依赖（输入） | 下游支撑（输出） | 同级互补 |
|------------------|-----------------|---------|
| **OV-2**（Needline → 细化） | → SV-2（系统资源流规格） | OV-2（上游概览） |
| OV-5b（活动模型） | → SvcV-2（服务资源流规格） | OV-5b（活动上下文） |
| DIV-2（资源元素定义） | → DIV-3（物理数据格式） | OV-1（场景背景） |

### OV-3 在作战资源流体系中的位置

```
作战视点的资源内容模型:
┌─────────────────────────────────────────┐
│                                         │
│  OV-2 (资源流描述) ──→ Needline 拓扑     │
│         ↓ 提取细化                       │
│  OV-3 (资源流矩阵) ──→ 资源属性详情      │  ← 本视图
│         ↓ 数据定义                       │
│  OV-5b (活动模型)  ──→ 活动间的资源流向  │
│         ↓ 信息实体                       │
│  DIV-2 (逻辑数据)   ──→ 资源元素的形式化  │
│                                         │
└─────────────────────────────────────────┘
```

> OV-3 是"作战视点中处理资源内容的模型套件之一"（其他包括 OV-2、OV-5b 和 DIV-2）。
