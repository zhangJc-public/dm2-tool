# SV-4: Systems Functionality Description（系统功能描述）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：每个系统**做什么功能**？输入输出数据流是什么？

SV-4 是系统视点的**行为/功能模型**——与 SV-1（结构）互补，详细描述系统的功能分解和数据流动。它是 OV-5b 作战活动模型的**系统层实现对应物**。

核心问题集：
| 问题 | 回答 |
|------|------|
| 每个系统执行什么功能？ | 功能层级分解 |
| 功能之间如何通过数据连接？ | 数据流图 |
| 所有必需的输入都满足了吗？ | 完整性检查 |
| 人机交互在哪里？ | HCI/GUI 功能 |
| 作战活动和系统功能的映射？ | 通过 SV-5 追溯 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────┐
│          SV-4: 系统功能描述                   │
│   （行为模型 / SV-1 的功能对应物）             │
├──────────────────────────────────────────────┤
│                                              │
│   与 OV-5b 的实现关系:                        │
│   OV-5b (作战活动) ──实现──→ SV-4 (系统功能)  │
│   "What (业务)"              "How (系统)"     │
│                                              │
│   ⚠️ 不一定是一对一映射! → 需要 SV-5 追溯矩阵 │
│                                              │
│   两种表达形式:                                │
│   ├─ 分类法功能层次树 (Taxonomic Hierarchy)    │
│   │   └─ 树形结构, 适合能力采购场景            │
│   └─ 数据流图 (Data Flow Diagram)             │
│       └─ 功能 + 数据流箭头 + 数据存储          │
│                                              │
│   文档内容:                                   │
│   • 系统功能 (System Functions)                │
│   • 功能间的 Resource Flows                    │

│   • 外部数据源/汇点(架构范围内)                │
```

**关键定位**：
- **OV-5b 的系统层实现**（作战活动→系统功能的映射目标端）
- **SV-1 的行为对应物**（如同 OV-5b 对应 OV-2）
- **人机交互建模**——HCI/GUI 作为系统功能的一部分
- 支持基于能力的采购中的**功能建模**

## 二、核心内容解析

### 2.1 必要性说明

> SV-4 解决**人工和系统功能**问题。
>
> **主要目的**：
> 1. 开发清晰的**必要数据流描述**——每个资源的输入（消费）和输出（生产）
> 2. 确保**功能连通性完整**（即资源的所有必需输入都已满足）
> 3. 确保**功能分解达到适当的详细程度**

预期用途：
1. 任务工作流描述
2. **功能性系统需求识别**
3. 系统功能分解
4. 关联人工和系统功能

### 2.2 推荐表达形式

| 形式 | 适用场景 | 说明 |
|------|---------|------|
| **分类法功能层次树** | 能力采购 | 需要对特定能力关联的功能进行建模时特别有用 |
| **数据流图（DFD）** | ⭐ 通用默认 | 功能 + 数据流箭头 + 数据存储 |
| **带泳道的 DFD 变体** | 多资源/多方案对比 | 强大的可视化差异技术 |

### 2.3 关键数据元素

| 元素 | 来源映射 |
|------|---------|
| System Functions（系统功能） | 来自 SV-4 自身；可叠加到 SV-1 |
| Resource Flows between Functions | → SV-6 详细记录 |
| Internal System Data Stores/Repositories | → DIV-3 物理来源 |
| External Sources/Sinks（架构范围内） | 外部交互边界 |

### 2.4 完整细节

#### 2.4.1 范围选项

| 范围类型 | 说明 |
|---------|------|
| **能力范围** | 不考虑哪个资源执行哪个功能 |
| **资源特定** | 聚焦特定资源的功能 |
| **内部数据流焦点** | 资源内的数据流动 |
| **外部数据流焦点** | 跨资源的数据交换 |
| **仅功能分配** | 只将功能分配给资源 |

#### 2.4.2 与 OV-5b 的非一对一映射

> 这些功能可能与 OV-5a 中捕获的 Operational Activities 相关。虽然 **Operational Activity Model (OV-5b)** 和 SV-4 的功能层次之间存在相关性，但它**不需要是一对一映射**——因此需要 **Function to Operational Activity Traceability Matrix (SV-5)** 来提供该映射。

#### 2.4.3 人机交互建模

⚠️ **重要扩展**：Systems 不仅限于内部系统功能，还可以包括：

- **HCI 和 GUI 功能**
- 消费或产生系统数据的功能

> 外部系统数据生产者或消费者可用于**表示与系统交互的人**。外部系统数据源/汇点（代表人或系统）与 HCI、GUI 或接口功能之间的 System Resource Flow 可用于表示**人机交互或系统-系统接口**。

#### 2.4.4 标准约束

> 适用于系统功能的标准（如 HCI 和 GUI 标准）也在开发此模型期间指定（并记录在 **StdV-1** 中）。

#### 2.4.5 泳道 DFD 变体（强大的方案对比工具）

> SV-4 Data Flow 模型的**图形变体可与泳道一起使用**。系统泳道可以关联到：
>
> 1. **一个 System**
> 2. **一组 Capabilities 和 System Functions**（通常基于 Physical Asset）
> 3. **一个执行 Activity 的 Performer**
>
> 泳道可以垂直或水平呈现。功能可以放置在与分配该功能的 System、Resources 或 Performer 关联的泳道中。这提供了一种以**功能术语**展示 Systems 或 Capabilities 之间交互的图形方式（这些通过 SV-1 上的系统连接显示）。
>
> 🎯 **这是一个用于可视化不同替代方案选项之间差异的强大技术**（这些选项可能有一组共同的功能）。

### SV-4 vs SvcV-4 vs OV-5b 三者对比

| 维度 | OV-5b | SvcV-4 | SV-4 |
|------|-------|--------|------|
| **视点** | 作战(OV) | 服务(SvcV) | 系统(SV) |
| **抽象层级** | 业务逻辑 | 服务逻辑 | 系统实现 |
| **独立性** | 实现无关 | 部分实现相关 | 实现相关 |
| **人工因素** | 角色抽象 | Personnel Type | HCI/GUI 功能 |
| **下游追溯** | → SvcV-4 + SV-4 | → SV-5 | → SV-5 |
| **互为对应** | — | OV-5b 的服务实现 | OV-5b 的系统实现 |
