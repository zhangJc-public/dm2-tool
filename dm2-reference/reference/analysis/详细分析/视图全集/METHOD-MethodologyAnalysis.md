## 二、模型分类 (Model Categories, pp.236-237)

### 2.1 六大模型类型

DoDAF 将所有描述性模型(Described Models) 归入以下六类：

| 类型       | 英文         | 定义                 | 典型视图                                 |
| -------- | ---------- | ------------------ | ------------------------------------ |
| **表格型**  | Tabular    | 行列结构化数据，含结构化文本     | OV-3, SV-3, SV-5a/b, SvcV-3a/b       |
| **结构型**  | Structural | 描述架构的结构性方面（拓扑/组成）  | OV-2, SV-1, SvcV-1                   |
| **行为型**  | Behavioral | 描述架构的行为性方面（动态交互）   | OV-5b, OV-6abc, SV-10abc, SvcV-10abc |
| **映射型**  | Mapping    | 两类信息之间的矩阵映射关系      | CV-5, CV-6, SV-5a/b, SvcV-5          |
| **本体型**  | Ontology   | 扩展 DoDAF 本体以适配特定架构 | DIV-1/2/3 (数据本体)                     |
| **图示型**  | Pictorial  | 自由形式图片             | **OV-1** ⭐                           |
| **时间线型** | Timeline   | 描述架构的程序性/时间维度方面    | PV-2, StdV-2, SV-9, SvcV-9           |

### 2.2 统计发现（来自官方调研）

> **"OV-1 是最常创建的产品，92% 的项目都会制作。SV-7 是最少创建的，仅 5%。"**
> — DoDAF Product Development Questionnaire Analysis Report

**关键洞察**：

| 排名  | 视图       | 创建率     | 解读                      |
| --- | -------- | ------- | ----------------------- |
| 🥇  | **OV-1** | **92%** | 每个项目都需要一张"门面图"——高层沟通刚需  |
| 🥉  | **SV-7** | **5%**  | 系统度量是最后才考虑的——多数项目不做性能基线 |
| -   | **无一可删** | 100%必要  | 调研结论：52个视图每个都有存在理由      |

## 三、架构分层 (Levels of Architecture, p.238)

### 3.1 Zachman 分层覆盖

DoDAF 使用 **Zachman Framework** 的分层概念帮助决策者确定需要哪些模型：

| Zachman 层次 | 对应决策者 | 关注点 | 典型 DoDAF 视图 |
|-------------|-----------|-------|----------------|
| **L1: Planner/规划者** | 高层战略决策者 | "我们为什么需要这个？" | CV-1, AV-1, **OV-1** |
| **L2: Owner/拥有者** | 业务流程负责人 | "业务怎么运转？" | OV-2, OV-5ab, OV-3 |
| **L3: Designer/设计者** | 系统架构师 | "系统如何构建？" | SV-1, SV-4, SvcV-1/2/4 |
| **L4: Builder/建造者** | 开发者/集成者 | "具体怎么实现？" | SV-2/3/5/6, SvcV-3/5/6 |
| **L5: Subcontractor/分包商** | 供应商/外包方 | "交付什么组件？" | StdV-1, PV-1/2/3 |

### 3.2 核心原则

> **⚠️ DoDAF 不强制规定任何层次必须制作哪些模型。** 上表仅为参考输入，实际选择取决于架构目的。

**实践意义**：
- 小型项目可能只需 L1+L2 层的视图
- 大型国防/政府项目需要全部五层
- **安全架构通常至少需要 L1~L4**（等保/密评要求从战略到实现的完整链路）

### 4.2 问询 → DM2 数据组 → 视图的高层映射

| 问询             | DM2 数据组                            | 回答此问题的核心视图                | **主驱动 SE 算子**                     |
| -------------- | ---------------------------------- | ------------------------- | --------------------------------- |
| **WHO (谁)**    | Performer / Org / Individual       | OV-4, CV-5, SvcV-3a       | **Cynefin 域判定** — 识别组织协作复杂度       |
| **WHAT (什么)**  | Resource / Data / Information      | DIV-1/2/3, OV-3, SV-6     | **TOC 约束理论** — 识别瓶颈资源与信息缺口        |
| **WHERE (哪里)** | Location / SpatialWholePart        | OV-2 (含地理标注)              | **OODA (O)** — 识别地理/网络空间对决策的影响    |
| **WHEN (何时)**  | TemporalPart / Time / Project      | PV-2, OV-4 (时间切片), SvcV-8 | **OODA (D/A)** + 模拟演化 — 时序韧性测试    |
| **WHY (为什么)**  | Rule / Goal / Measure / Capability | CV-1/2, OV-6a, SV-10a     | **StdV 强制对撞 + TRIZ** — 指向冲突求解     |
| **HOW (怎么做)**  | Activity / Function / Service      | OV-5b, SV-4, SvcV-4       | **溯因推理 (Abduction)** — 从意图到动作的逻辑链 |

> 📝 **关于"主驱动"的说明**: 每个 6W 维度标注的是**主要驱动算子**，非排他性。实际架构问题通常需要多算子协同（如一个 Why 问题可能同时需要溯因推理补全因果链 + StdV 对撞检查标准合规）。四算子平权，不设优先级。—— [来源: AAI 附录 2.1]

### 4.4 三大结构性发现

基于以上 55 条映射，可以得出原始 DoDAF 规范中隐含但未明确表述的三条结构规律：

#### 发现 1：SvcV 是唯一的「全能桥梁」📊 [统计推导]

> **推导依据**: 基于 §4.3 的 55 条 6W 视图映射矩阵逐条统计得出。此结论为 AAI 分析团队的统计发现，非 DoDAF 规范原文声明。

| 视点       | 出现在 6W 中的次数  | 定位                         |
| -------- | ------------ | -------------------------- |
| **SvcV** | **6/6 全部出现** | 全能桥梁 —— 任何情报提问都必须检查服务契约一致性 |
| OV       | 5/6          | 业务核心 —— 缺失 Project 视点      |
| SV       | 5/6          | 技术核心 —— 缺失 Project 视点      |
| CV       | 2/6          | 战略锚点 —— 仅出现在 What/Why      |
| PV       | 1/6          | 执行载体 —— 仅出现在 When          |
| DIV      | 1/6          | 数据基础 —— 仅出现在 What          |
| StdV     | 1/6          | 法理判决 —— 仅出现在 Why           |

> **实践含义**: 服务化架构（SOA/SaaS）不是技术选择，而是 DoDAF 元模型的结构必然。SvcV 在 6W 中的全覆盖意味着**服务是唯一能同时承载 Who/What/Where/When/Why/How 六维信息的架构抽象**。

#### 发现 2：Why → StdV 的「法理链条」

官方矩阵规定 **Why（动机/规则）** 是唯一触及 **Standards (StdV)** 的维度。这意味着：
- 当架构师回答"为什么需要这个系统"时，**必须同时回答"遵循哪些标准"**
- StdV 不再是"附录式的合规检查表"，而是**因果分析的逻辑终点**
- 在 L2 认知映射层，StdV 充当"三维对撞结果的法理判决书"

```
Why (动机查询)
  ├─→ CV-1~7 (能力证明): "我们能做到什么"
  ├─→ OV-6a / SV-10a / SvcV-10a (规则定义): "受限于什么"
  └─→ StdV-1 / StdV-2 (标准剖面): "⭐ 法理判决 —— 合规即动机"
```

#### 发现 3：What/Who/When → PV 的「执行闭环」

官方矩阵规定 **Who、What、When** 都指向了 **Project (PV)**。这意味着：
- 当你提到某个资源缺口（What）、某类执行者（Who）、某个时间节点（When）
- AI 必须**自动激活 PV 视点**来显式化项目依赖
- 这是 **TOC 约束理论** 在 DoDAF 中的自然落脚点：瓶颈必须在 PV 中被看见

#### 发现 4：CV-1 → OV-1 的「1:n 拆分规则」🔴 强制检查点

> **从 P0-OV-1 分析报告（§1.2 + §3.1）提取，已作为强制检查点固化。**

```
CV-1（单一战略愿景）
  │
  │ 1 : n
  │
  ├── 时间轴维度 → OV-1a(基线 As-Is) / OV-1b(目标 To-Be) / OV-1c(过渡)
  ├── 业务域维度 → OV-1d(域A场景) / OV-1e(域B场景) / ...
  └── 关注方维度 → OV-1f(业务视角) / OV-1g(安全视角) / OV-1h(技术视角)
```

| 维度      | 拆分条件         | 示例                |
| ------- | ------------ | ----------------- |
| **时间轴** | 项目有明确基线/目标差异 | 政务平台：孤岛现状 vs 统一平台 |
| **业务域** | 覆盖 ≥2 个独立任务域 | 护卫舰：反潜 / 防空 / 反舰  |
| **关注方** | 不同干系人关注面不同   | 安全侧威胁模型 vs 业务侧数据流 |

**🔴 强制检查清单（生成 OV-1 前逐条回答）：**

| #   | 检查问题                   | 如果"是"则必须拆分  |
| --- | ---------------------- | ----------- |
| C1  | 是否有 As-Is vs To-Be 差异？ | 至少 2 张 OV-1 |
| C2  | 是否涉及 ≥2 个独立业务域？        | 按域各 1 张     |
| C3  | 是否有安全/合规独立视角？          | 补充安全域 OV-1  |
| C4  | 不同干系人能否用一张图满足？         | 按干系人拆分      |

> ⚠️ **默认陷阱**: 大模型生成天然倾向「一份输入→一份输出」的线性模式。**必须用此检查清单打破默认值。**

### 4.5 实战示例：物流事务流程改造

规范原文给出了一个精妙的示例：

> 决策：改变一个物流事务流程（一组活动的复合体）
>
> **How**: 流程已文档化 → 含性能度量、所需服务、能力支持
>
> **What**: 执行流程所需的数据（并发收集）
>
> **Where**: 执行地点的位置和行政信息
>
> **Who**: 行为的执行者
>
> **When**: 所需的时间框架
>
> **Why**: 规则、目标和预期结果（影响性能度量）

**这个例子揭示了 DoDAF 方法的本质：不是"画完52张图"，而是"用最少的视图回答完整的6W1H问题"。**

结合 AAI 的 SE 算子视角，这个物流示例可以进一步解读为：

| 问询    | DoDAF 原文   | SE 算子增强解读                           |
| ----- | ---------- | ----------------------------------- |
| How   | 流程已文档化     | **溯因推理**：从"改变流程"意图反推所需的活动分解树(OV-5a) |
| What  | 并发收集数据     | **TOC 瓶颈识别**：数据资源缺口是否是系统产出瓶颈？       |
| Where | 地点和行政信息    | **OODA 定位**：地理约束是否影响决策速度？           |
| Who   | 行为执行者      | **Cynefin 判定**：人员协作模式属于哪种复杂度域？      |
| When  | 时间框架       | **模拟演化**：关键路径上是否有单点故障风险？            |
| Why   | 规则/目标/预期结果 | **StdV 对撞**：目标是否与现有标准冲突？            |

### 4.6 DM2 概念数据模型上的问询叠加

规范在 p.240 明确展示了 **6 大问询维度直接叠加到 DM2 CDM（概念数据模型）上**：

```
DM2 Conceptual Data Model:
+-----------------------------------------------------+
|                                                     |
|  [Data Description] <- WHAT                         |
|       ^                                             |
|  [Function Description] <- HOW (+Performer/Measure)  |
|       ^                                             |
|  [Network Description] <- WHERE                     |
|       ^                                             |
|  [People Description] <- WHO (DM2含Organization)     |
|       ^                                             |
|  [Time Description] <- WHEN                        |
|       ^                                             |
|  [Motivation Description] <- WHY (含Capability需求)   |
|                                                     |
+-----------------------------------------------------+
```

这意味着：**DM2 不是独立于视图的存在——它是 6W1H 问询的本体论基础。**

### 4.7 核心实体的三位一体对应

> **来源**: AAI 附录 2.2 核心实体

核心实体（Core Entities）是连接"你想什么"与"系统是什么"的**最小逻辑单位**，直接源于 DoDAF 2.02 DM2 CDM。官方将复杂系统工程世界抽象为 **13 个顶层核心实体**，其中最关键的几个承载了 6W 疑问的全部语义。

#### 三层映射体系

| 映射层         | 对应关系             | 示例                                              |
| ----------- | ---------------- | ----------------------------------------------- |
| 语义层 → 6W 疑问 | 核心实体是 6W 的唯一法定答案 | Who→Performer, What→Resource/Data, How→Activity |
| 物理层 → 工程对象  | 核心实体是现实物体的数字孪生体  | System→国产加密芯片, Activity→实人认证流程                  |
| 视图层 → 52 视图 | 核心实体是视图的公共零件     | Activity 同时支撑 OV-5b / SV-4 / SvcV-4             |

#### 核心实体高分辨率表

| DoDAF 核心实体 | 6W 疑问 | 现实对象示例 | SE 理论价值 |
|---------------|---------|-------------|-----------|
| Resource (资源) | What | 硬件、软件、人员、能量 | TOC 瓶颈识别的基准对象 |
| Activity (活动) | How | 任务、操作、处理过程 | 服务化解耦的最小颗粒度 |
| Performer (执行者) | Who | 组织、系统、岗位、角色 | OODA 循环的决策主体 |
| Capability (能力) | Why | "在 2 秒内完成隔离比对" | 战略评估的量化指标 |
| Condition (条件/规则) | Why | 公安隔离标准、金融时延红线 | StdV 对撞的逻辑约束 |
| Location (位置) | Where | 地理坐标、网络节点、机房 | 分布计算的空间约束 |

#### 为什么强调核心实体？

1. **确定性**: 每个 DM2 实体在 PES 中都有唯一的 XML 标签
2. **可计算性**: SE 算子（如 TOC）只能作用于有属性的实体，不能作用于线条
3. **智能化基础**: 大模型通过识别核心实体，才能在推理层利用 TRIZ 针对具体实体进行冲突求解
4. **视图联动**: 修改一个实体的属性，所有引用该实体的视图自动同步——解决"视图打架"

> **一句话总结**: 核心实体是 DoDAF 官方元模型的**本体（Ontology）**，架构中所有逻辑与物质的**共同契约**。从"画图"到"建模"的质变，关键在于是否以核心实体为中心组织架构描述。

## 六-B、SE 理论算子体系（AAI 智能增强） ⭐⭐⭐

> **本节完全来自 [AAI 附录 2.1 SE 理论](../../../政事/研制/架构即智能/架构即智能概要设计.md#附录21-se)**，是原始 DoDAF 方法论**完全没有涉及但至关重要的方法论引擎**。
>
> 传统 SE（NASA / INCOSE）侧重静态 V 模型或需求工程，而 AAI 面向的是"复杂性和架构主动性"。以下四大核心算子的集成构成了 L2 认知映射层的推理引擎。

### 5B.1 四大 SE 算子总览

| 算子                   | 解决的问题   | 在架构方法中的角色                  | 对应 6W      |
| -------------------- | ------- | -------------------------- | ---------- |
| **Cynefin 框架**       | 感知（看哪里） | 进气格栅——判定问题复杂度 → 决定调用多少视图   | Who        |
| **溯因推理 (Abduction)** | 因果（为什么） | 逻辑胶水——补全 Why 与 How 之间的隐含链条 | Why→How    |
| **OODA 循环**          | 韧性（会怎样） | 神经中枢——让静态架构图动态演化           | Where+When |
| **TOC 约束理论**         | 落地（怎么做） | 生存法则——锁定视图丛林中的最弱一环         | What→PV    |

### 5B.2 Cynefin 框架：认知分类算子

**用途**: 解决"感知"问题——防止在简单问题上过度建模，在复杂问题上建模不足。

```
问题输入 → Cynefin 判定
    │
    ├─ Clear (明晰)   → 最小视图集 (OV-1 + CV-1)
    ├─ Complicated (繁杂)  → 标准视图集 (P0 核心 17 份)
    ├─ Complex (复杂)      → 全量视图集 (P0+P1 = 38 份) + 行为三件套
    └─ Chaotic (混沌)      → 全量 + Fusion Views + 实时模拟
```

**在 DoDAF 中的具体应用**:
- 公安/金融跨域项目通常落在 **Complex** 域 → 需要 OV/SV/SvcV 三视点全覆盖
- 单系统升级通常在 **Complicated** 域 → OV + SV 即可
- 应急响应场景可能进入 **Chaotic** 域 → Dashboard + Fusion View 实时驱动

### 5B.3 OODA 循环：动态博弈算子

**用途**: 解决"韧性"问题——通过"观察-定向-决策-执行"让静态架构图动起来。

```
传统 DoDAF:  需求 → 架构 → 文档 (一次性产物)
OODA 增强:   Observe → Orient → Decide → Act (持续循环)
                    ↑         │
                    └─────────┘
                 架构图随态势演化
```

**与 DoDAF 视图的绑定**:
| OODA 阶段 | 对应 DoDAF 视图 | 注入的异常变量示例 |
|-----------|---------------|------------------|
| **Observe** | OV-6c (时序), SV-10c (追踪) | 网关延迟 spike、认证失败率上升 |
| **Orient** | OV-2 (节点), OV-3 (信息交换) | 流量模式变化、新接入节点 |
| **Decide** | OV-6a (规则), SV-10a (系统规则) | 启用降级策略、切换备用链路 |
| **Act** | SV-10b (状态转移), PV-2 (时序) | 执行熔断、通知运维 |

### 5B.4 TOC 约束理论：资源瓶颈算子

**用途**: 解决"落地"问题——在 52 个视图的数据丛林中只盯住制约系统产出的"最弱一环"。

```
TOC 核心逻辑:
  任何系统至少存在一个瓶颈约束 → 识别它 → 挖尽它 → 迁移到下一个

DoDAF 映射:
  Gap Matrix (CV 能力差距)  ← TOC: "哪里不足？"
       ↓
  SV-7 / SvcV-7 (度量矩阵)   ← TOC: "约束有多紧？"
       ↓
  PV-2 (项目时序)            ← TOC: "何时释放？"
       ↓
  CV-3 (能力演化)            ← TOC: "释放后提升多少？"
```

**关键洞察**: 当你提到某个资源缺口（What），L2 必须自动激活 **PV 视点**显式化项目依赖。这就是 TOC 在 DoDAF 中的自然落脚点。

### 5B.5 溯因推理 (Abductive Reasoning)：逻辑缝合算子

**用途**: 解决"因果"问题——推测专家没说出口的潜规则或隐含约束。

```
演绎 (Deductive):  规则 + 事实 → 必然结论 (如: 所有A都是B, x是A, 所以x是B)
归纳 (Inductive):  观察 → 一般规律 (如: 天天日出 → 太阳总是从东方升起)
溯因 (Abductive):  结果 → 最佳解释 (如: 草地湿了 → 最可能下过雨)

在架构中的应用:
  专家意图: "我们要做一个安全隔离系统"
       ↓ 溯因推理
  AI 补全:
    - 隐含规则: 需符合等保三级 (Why → StdV-1)
    - 隐含资源: 需要国产加密卡 (What → SV-7)
    - 隐含执行者: 需要密钥管理员角色 (Who → OV-4)
    - 隐含时序: 密钥定期轮换 (When → PV-2)
```

### 5B.6 四算子的闭环运行模型

四个算子构成完整的**闭环系统化思考模型**：

```
        ┌─────────────────────────────────────┐
        │                                     │
        │  ① Cynefin: 看哪里                  │
        │     (确定范围 → 选择视图集)           │
        │              │                       │
        │              ▼                       │
        │  ② Abduction: 为什么                │
        │     (构建因果 → 补全隐含约束)        │
        │              │                       │
        │              ▼                       │
        │  ③ OODA: 会怎样                      │
        │     (模拟动态 → 测试时序韧性)         │
        │              │                       │
        │              ▼                       │
        │  ④ TOC: 怎么做                       │
        │     (锁定执行 → 识别瓶颈资源)         │
        │              │                       │
        │              └───────→ ① (循环)       │
        │                                     │
        └─────────────────────────────────────┘
```

> **平权原则**: 四算子**不设置更高权重，维持平权驱动**。避免算法偏见（例如为了 OODA 快而牺牲安全标准的 Why 约束）。这是 AAI 的关键设计决策。

## 八、呈现技术 (Presentation Techniques, pp.243-256) ⭐⭐⭐

这是整个方法论章节**篇幅最长、最实用**的部分。

### 7.1 核心问题意识

> **"许多组织开发了强制性的架构产品，却使其变成了昂贵的架子上积灰品（expensive shelf-ware），而不是用它向利益相关者传达重要、准确、相关的信息。"**
> — DoDAF Vol.II p.243

**根本矛盾**：架构师能理解复杂的技术信息，但**决策者不能**。

### 7.2 Information Bridge（信息桥梁）概念

```
┌─────────────┐              ┌─────────────┐              ┌─────────────┐
│   Architect │  ──桥梁──→   │  Presentation │  ──沟通──→   │   Decision  │
│ (技术信息)  │  (翻译/重铸) │  Techniques  │  (易懂形式)   │    Maker    │
└─────────────┘              └─────────────┘              └─────────────┘
                                   │
                        图形/文本/表格/仪表盘
                     (符合组织文化的表达方式)
```

### 7.3 决策者层级与呈现策略

```
Level 1: Planner (规划者)      Level 2: Owner (业务拥有者)
├─ 喜欢图形化的墙图(wall chart)  ├─ 需要业务流程级别的信息
├─ 战略层面的宏观视角             ├─ 关注自己的业务域及直接关联
└─ OV-1 / CV-1 / PV-1 最有效    └─ OV-2 / OV-5 / CV-3 最有效

Level 3: Designer (设计者)      Level 4: Builder (建造者)
├─ 需要系统架构级的技术细节       ├─ 需要实现和使用的具体细节
├─ SV-1 / SV-4 / SvcV-2 最有效  ├─ SV-2 / SV-6 / StdV-1 最有效
└─ Composite Views 非常有价值     └─ 可能需要原始规格说明

Level 5: Subcontractor (分包商)
├─ 需要不同粒度的技术数据和任务信息
├─ 具体的接口协议、数据格式
└─ SV-2(详细) / StdV-1 / PV-3 最有效
```

### 7.4 五大呈现技术

#### 7.4.1 Composite Views（组合视图）🥇 最推荐

**定义**: 将多份架构数据以与**特定决策者相关**的格式组合展示。

**特点**:
- 从多个数据源抽取信息，形成整体视角
- 支持快照对比（As-Is vs To-Be 并排展示）
- 可用三维方式绑定数据片段
- 由**相互支持的架构视图组成**（如 SV-4 系统函数支持 OV-5 活动）

**适用受众**:
- **Process Owner**: 有技术监督能力的流程负责人
- **Designer**: 需要实现细节的设计者
- **Builder**: 需要使用方法的系统架构师

**示例**: 活动"Determine Accession Type"通过 UI 由系统函数"Maintain Candidate Data"支撑，涉及数据元素"Accession Type Information"和"Other Candidate Information"，由"Human Resource Specialist"执行——**将 OV-5/SV-4/DIV/OV-4 四个视图的信息整合到一张图中**

**中国场景**: 等保测评中的"整体架构图"就是典型的 Composite View——把网络拓扑(SV-1)、安全区域划分(OV-2)、安全策略(OV-6a)、数据流(SV-6) 合成为一张总图。

#### 7.4.2 Dashboard Views（仪表盘视图）

**定义**: 为给定业务上下文**聚合抽象后的架构信息**，面向特定利益相关者。

**构成要素**: 状态(status)、趋势(trend)、偏差(variance to plan/forecast/budget)

**优势**:
- 高管熟悉这种形式（其他业务领域广泛使用）
- **一目了然**地管理绩效目标
- 友好的用户界面

**可视化技术选择**:

| 技术 | 适用场景 | 注意事项 |
|------|---------|---------|
| **饼图 Pie Chart** | ≤6 个元素的占比关系 | 人眼难以分辨角度差异；不适合精确对比 |
| **柱状图 Bar Chart** | 分类比较/序列对比 | 注意离群值(outlier)会压缩其他数据的可视空间 |
| **折线图 Line Chart** | 时间趋势分析 | 多系列堆叠适合趋势对比 |
| **面积图 Area Chart** | 多系列趋势+总量感 | 折线图的变体 |
| **表格/列表 Table/List** | 大量非数值数据/复杂关系 | 无法编译为视觉工具时使用 |

**关键原则**: Dashboard 展示的是**高度聚合和抽象的信息**，主要面向高级决策者。

#### 7.4.3 Fusion Views（融合视图）

**定义**: 类似 Composite View，但**额外包含架构描述之外的外部信息**。

**与 Composite View 的核心区别**:

| | Composite View | Fusion View |
|--|---------------|-------------|
| **数据来源** | 仅限架构描述内部 | 架构描述 **+ 外部数据** |
| **典型外挂数据** | 无 | 资金信息、成本数据、WBS 进度、市场情报 |
| **敏感性** | 一般可公开 | **可能包含敏感信息**，限定查看者 |
| **用途** | 解释架构内部关系 | 弥合 EA 分析与其他分析的鸿沟 |

**典型应用场景**:
- 项目投资决策：架构方案 + 成本估算 + ROI 分析
- 能力差距评估：当前能力(CV) + 预算约束(PV) + 外部威胁情报
- 供应商选择：技术方案(SV/SvcV) + 市场报价 + 交付风险评估

**受众**: 全部团队成员（Planner/Owner/Designer/Builder/Subcontractor）

#### 7.4.4 Graphics Views（图形视图）

**定义**: 使用图片/地图/图表等形式对数据进行**视觉化表示和操作**。

**特点**:
- 面向**视觉型学习者**
- 缺乏底层复杂性 → 更**抽象**
- 通常面向**高层受众**
- 人们理解图片的速度远快于文字或模型文档

**关键警告**:
> **"如果利益相关者不知道信息从何而来，或者他们对详细的架构信息缺乏信心，那么图形视图对他们来说基本上没有意义。"**
> — p.253

**必须强调底层架构信息的来源**——OV-1 的每一部分都应对应到详细的业务区域，由复杂的架构视图集支撑。又或者每一个OV-1都对应到详细的业务区域，由复杂的架构视图支撑。

**DoDAF 中典型的 Graphics Views**: **OV-1**（高层作战概念图）和 **OV-2**（高层连接描述）

#### 7.4.5 Reference Models（参考模型）

**定义**: 对底层架构数据进行**文本化提取**，将元素转化为文字框架。

**FEA 示例**: 美国 FEA（联邦企业架构）的五参考模型:

| 参考模型 | 缩写 | 内容 |
|---------|------|------|
| Performance Reference Model | PRM | 性能度量框架 |
| Business Reference Model | BRM | 业务运营分类法（39个业务线） |
| Service Component Reference Model | SRM | 服务组件分类 |
| Data Reference Model | DRM | 数据共享标准 |
| Technical Reference Model | TRM | 技术标准分类 |

**BRM 结构详解**:
- 四大领域: Services for Citizens / Mode of Delivery / Support Delivery / Management of Government Resources
- 39 条业务线(Lines of Business)
- 每条业务线下设子功能(Sub-functions) = BRM 最细粒度
- 例: Environmental Management → (1) Monitoring & Forecasting (2) Remediation (3) Pollution Prevention & Control

**中国对应物**: 
- **GB/T 4754 国民经济行业分类** ≈ BRM
- **政务目录体系** ≈ BRM 的政务版
- **信创分类图谱** ≈ TRM 的中国版

## 九、配置管理 (Configuration Management, pp.257-258)

### 8.1 CM 的两大范畴

| 范畴 | 内容 | 权威机构 |
|------|------|---------|
| **CM Guidance** | 对 DoD 内按 DoDAF 制作的架构描述实例的配置管理指导 | 各项目经理遵循 |
| **Framework CM** | DoDAF 框架文档本身的配置管理 | **DoD CIO**, OASD 企业架构与标准局 |

### 8.2 遵循 CM 的收益（6 项）

1. **DM2 三层统一利用**（Conceptual + Logical + PES）：数据收集、组织、存储、文档化的一致性
2. **技术指导遵循**：基于架构数据和期望视点的视图创建和图形表示
3. **公共词汇和语法**：促进 DoD 组件间的共同理解，确保互操作性
4. **需求可追溯性**：架构数据更容易关联到基线需求，变更影响可评估
5. **变更评估依据**：已知基础上的变更 + 所选视角下的有效性评估手段
6. **联邦制支持**：个体架构描述在更高层企业视图中的联邦集成

### 8.3 与现有 DoD CM 过程的关系

CM 活动与以下三者的过程互补：
- **DARS** (Defense Acquisition Regulation System)
- **DISR** (DoD IT Standards Registry)
- **MDR** (Metadata Registry)

## 十、方法论的中国适配总结

### 9.1 直接可用部分

| 方法论要素 | 中国适用性 | 说明 |
|-----------|----------|------|
| **5W1H 问询矩阵** | ⭐⭐⭐⭐⭐ | 通用性强，无需修改即可用于任何中国架构项目 |
| **模型六分类** | ⭐⭐⭐⭐ | 分类逻辑普适，但"Pictorial"在中国实践中更常用 |
| **Composite Views** | ⭐⭐⭐⭐⭐ | 等保/密评报告的总图即为此类，高度契合 |
| **Dashboard Views** | ⭐⭐⭐⭐ | 运维监控/态势感知平台即是 Dashboard 的变体 |
| **Zachman 分层** | ⭐⭐⭐⭐ | 国内 EA 实践也普遍接受分层思想 |
| **CM 六项收益** | ⭐⭐⭐⭐ | 国标 GB/T 8566/ITIL CM 思想一致 |

### 9.2 需要本地化部分

| 方法论要素 | 适配方向 | 建议 |
|-----------|---------|------|
| **Reference Models** | FEA BRM → 中国政务目录/国标行业分类 | 建立中国版参考模型库 |
| **Fusion Views** | 增加"合规信息"作为外挽数据维度 | 融入等保/密评/关基测评结果 |
| **Primitives (AMP)** | 建议建立中文版 CN-AMP | 基于 DM2 + UML + BPMN + 中文语境 |
| **决策者层级** | 增加"监管者/审计者"(Auditor)层级 | 中国特色——等保测评师/密码评测员 |
| **创建率统计** | 重新调研中国项目的视图使用频率 | 中国国情下 OV-1 和 SV-7 的比率可能不同 |

### 9.3 安全架构专用增强建议

基于方法论原理，提出安全架构领域的**增强型呈现技术栈**:

| 传统 DoDAF | 安全架构增强版 | 说明 |
|------------|--------------|------|
| Composite View | **Security Posture View** | 在组合视图基础上增加安全域着色(红/黄/绿) |
| Dashboard | **SOC Dashboard** | 融合 SIEM/SOAR 数据的实时态势感知 |
| Fusion View | **Compliance Matrix View** | 架构 + 等保要求/密评条款 + 差距标注 |
| Graphics View | **Threat Scenario Diagram** | OV-1 基础上叠加攻击路径(红色箭头) |
| Reference Model | **Security Control RM** | 以 GB/T 22239/GB/T 39786 为参照的安全控制参考模型 |
