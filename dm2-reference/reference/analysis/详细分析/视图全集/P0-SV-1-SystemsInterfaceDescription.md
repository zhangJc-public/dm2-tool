# SV-1: Systems Interface Description（系统接口描述）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：**系统**之间如何组合和交互来实现作战需求？

SV-1 是系统视点的**结构模型**——展示系统、资产和人员如何配置成满足特定能力的方案。它与 SvcV-1 平行但关注点不同。

核心问题集：
| 问题 | 回答 |
|------|------|
| 有哪些系统及子系统？ | 系统分解结构 |
| 系统之间有什么接口？ | System Resource Flows |
| 谁使用这些系统？ | 组织资源和人员类型 |
| 部署在哪些物理平台上？ | 平台/设施位置 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────────┐
│          SV-1: 系统接口描述                       │
│   （系统结构 / OV-2 Needlines 的系统实现）        │
├──────────────────────────────────────────────────┤
│                                                  │
│   OV-2 (Needline) ──实现(Realize)──→ SV-1       │
│                                                  │
│   ┌─────────┐   Resource Flow   ┌─────────┐      │
│   │System A  │ ────────────────→ │System B  │      │
│   │(Source)  │   (point-to-point) │(Target)  │      │
│   └────┬─────┘                   └─────────┘      │
│        │                                           │
│   部署在:                                         │
│   ┌────┴────┐                                     │
│   │ Platform │ ← Physical Asset                  │
│   └─────────┘                                     │
│   使用者: Organization / Personnel Type            │
│                                                  │
│   与 SvcV-1 的关键差异:                            │
│   ├─ SV-1:    关注 System-to-System 点对点接口      │
```

**关键定位**：
- **OV-2 Needlines 的系统层实现**
- **架构权衡研究的基础**——逻辑/物理分离使得可基于风险/成本/可靠性等进行方案对比
- 与 SvcV-1 平行的**系统视点版本**

## 二、核心内容解析

### 2.1 必要性说明

> SV-1 解决系统的**组成和交互**问题。DoDAF V2.0 中，SV-1 将**人类元素作为 Performer 类型**纳入——组织和人员类型。
>
> SV-1 通过描绘资源如何结构和交互来**实现 OV-2 中指定的逻辑架构**，将作战和系统架构模型连接起来。
>
> ⚠️ 一个 OV-2 Needline 可能**转换为多个** System Resource Flows。

预期用途：
1. 系统概念定义
2. 系统选项定义
3. 系统 Resource Flow 需求捕获
4. 能力集成规划
5. 系统集成管理
6. 作战规划（能力和执行者定义）

### 2.2 两种互补使用方式（与 SvcV-1 相同模式）

| 方式 | 内容 |
|------|------|
| **方式一** | 描述架构中资源间交换的 Resource Flows |
| **方式二** | 以能力组件及其在平台和其他设施上的物理集成为基础描述**解决方案或解决方案选项** |

### 2.3 关键数据元素

| 元素 | 说明 |
|------|------|
| **Systems / Sub-systems** | 系统及子系统（可任意深度分解） |
| **Performers（Systems as）** | 系统作为执行者类型 |
| **Personnel Types** | 人员类型（V2.0 新增） |
| **Organizational Resources** | 组织资源 |
| **Physical Assets / Platforms** | 部署资源的物理资产 |
| **System Resource Flows** | 系统间的资源流动指示器 |
| **System Functions（可选叠加）** | 来自 SV-4 的功能 |
| **Operational Activities/Capabilities/Locations（可选标注）** | 来自 OV-2 的追溯 |

### 2.4 完整细节

#### 2.4.1 物理资源的组成规则（与 SvcV-1 相同）

> 贡献于能力的物理资源要么是组织资源，要么是**物理资产**——即**系统不能单独贡献**（它必须托管在由组织资源使用的物理资产上）。

#### 2.4.2 人类因素 + 架构权衡能力

> DoDAF V2.0 包括**人类因素**（作为 Personnel Types 和一种 Performer 类型）。如果架构师希望描述具有人工元素的 System，应使用 Systems、Personnel Types 和 Performers 将人工和系统元素包装在一起。

**OV/SV 分离的架构权衡价值**：

> 当 OV 套件指定一组需求时（作为特定的作战计划或采购场景），OV-2/OV-5a/OV-5b 指定**逻辑结构和行为**，SV-1/SV-4 指定**物理结构和行为**（达到架构利益相关者所需的详细程度）。这种逻辑和物理的分离提供了基于 DoDAF 模型中的架构内容进行**架构权衡研究的机会**。
>
> OVs 和 SVs 中的结构和行为模型使架构师和利益相关者能够**快速确定每个替代方案中由人执行的功能和由系统执行的功能**，从而可以基于风险、成本、可靠性等进行分析。

#### 2.4.3 从 OV-2 到 SV-1 的追溯

> SV-1 可以选择性地用 OV-2 中最初指定的 Operational Activities、Capabilities 和/or Locations 标注。通过这种方式，可以建立从逻辑 OV 结构到物理 System Viewpoint 结构的**追溯性**。
>
> 在许多情况下，OV-2 中描绘的作战活动和位置很可能是 SV-1 中显示的资源的**逻辑表示**。

#### 2.4.4 SV-1 与 SV-4 的互补关系

> SV-1 和 SV-4 提供**互补的表示（结构和功能）**。两者都可以先建模，但通常采用**迭代方法**逐步构建系统描述的详细程度。

## 三、关联视图

| 上游依赖（输入） | 下游支撑（输出） | 同级互补 |
|------------------|-----------------|---------|
| OV-2（Needlines → 实现） | → SV-2（Resource Flow 详情） | SvcV-1（服务接口对应物） |
| CV（能力需求） | → SV-3b（矩阵汇总） | SV-4（行为/功能对应物） |
| OV-5b（活动→系统追溯来源） | → SV-6（RF 矩阵） | |

### SV-1 与 SvcV-1 完整对比

| 维度 | SV-1 (系统接口) | SvcV-1 (服务接口) |
|------|----------------|-------------------|
| **核心焦点** | System-to-System 点对点接口 | Provider + Data（发布/订阅） |
| **接口约定** | Source ↔ Target 有明确约定 | 发布者/订阅者松耦合 |
| **适合模式** | 传统系统集成、协议绑定接口 | SOA、以网络为中心的数据战略 |
| **Resource Flow 含义** | 系统间的具体数据通道 | 服务间的逻辑数据供给 |
| **人类元素** | ✅ V2.0 新增 Performer | ✅ V2.0 新增 Performer |
| **OV-2 映射** | Needline → 多个 System RF | Needline → 多个 Service RF |
