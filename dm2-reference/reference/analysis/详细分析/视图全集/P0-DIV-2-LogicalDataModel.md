# DIV-2: Logical Data Model（逻辑数据模型）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这些业务信息在逻辑上如何结构化？（不涉及具体实现）

DIV-2 是从概念到物理的**中间桥梁**——它将 DIV-1 的业务信息实体形式化为**与技术无关的逻辑数据结构**。

核心问题集：
| 问题 | 回答 |
|------|------|
| 数据元素之间有什么逻辑关系？ | 形式化的数据结构 |
| 数据定义是否一致？ | 公共数据字典 |
| 业务需求对数据的约束是什么？ | 来自 OV-6a 的规则验证 |

### 1.2 在架构描述中的角色

```
┌──────────────────────────────────────────────┐
│          DIV-2: 逻辑数据模型                   │
│     （形式化数据结构 / 实现无关层）             │
├──────────────────────────────────────────────┤
│                                              │
│   DIV-1 (概念) ──理论映射──→ DIV-2 (逻辑)    │
```

## 二、核心内容解析

### 2.1 必要性说明

> DIV-2 允许在不考虑**实现特定或产品特定问题**的情况下分析架构的数据定义方面。
>
> 另一个目的是提供一个**公共数据定义字典**，以在任何包含逻辑级数据元素的描述中一致地表达模型。

### 2.2 推荐表达形式

官方明确指出 **不 endorsement 特定方法论**，选择取决于技术：

| 技术方向 | 推荐方法 |
|---------|---------|
| **关系型理论** | 实体关系图（ER Diagram） |
| **面向对象** | 类图 / 对象图（Class/Object Diagram） |
| **消息导向** | 引用消息格式标准 |
| **文件传递** | 文件格式描述 |

### 2.3 关键数据元素与关联

| 来源视图 | 到 DIV-2 的映射关系 |
|---------|-------------------|
| **OV-1** 高层作战概念图 | Data 可能是 Information 的**简单子类型**（过程化/结构化描述方式） |
| **OV-1** 或 **活动资源流**（Resource=Data） | 也可能是 **Information 和 Data 的整体-部分（及重叠）关系** —— 更复杂的关系模式 |
| **OV-6a** 作战规则模型 | DIV-2 的信息和数据元素可被其捕获的业务需求所约束和验证 |
| **OV-6c** 事件追踪描述 | DIV-2 还捕获连接生命线的消息的**信息内容** |
| **StdV-1/StdV-2** 标准/标准预测 | DIV-2 可捕获因标准要求而需要的元素 |

### 2.4 完整细节

#### 2.4.1 数据模型质量度量体系

官方提供了一个相当完整的**逻辑数据模型质量框架**：

**验证因子（Validation Factors）— 是否构建了正确的模型？**

| 因子 | 说明 |
|------|------|
| 信息需求保真度（Information Requirements Fidelity） | 模型是否准确反映了业务需求 |
| 概念/逻辑/物理可追溯性（Traceability） | 三层之间的追溯链是否完整 |
| 遵循政府和行业标准 | 符合性检查 |
| 领域值（Domain Values） | 取值范围是否合理 |
| 资源交换和互操作需求 | 互操作支持度 |
| 以网络为中心的因素 | XML 注册等 |

**验证因子（Verification Factors）— 是否构建得好？**

| 类别 | 具体因素 |
|------|---------|
| **设计因子** | 规范化程度、紧致性、抽象和泛化 |
| **本体论基础** | 语义纯度 |
| **冗余控制** | 逻辑和物理冗余 |
| **分离关注点** | 职责分离 |
| **软件质量因子** | 文档、命名约定、命名和业务语言、定义完整性、一致性、可实现性、枚举/自由文本比 |

#### 2.4.2 规范化（Normalization）要点

> 一个设计因子的例子是**规范化**——本质上每个现实世界对象只有一种表示。规范化有不同程度，**第三范式（3NF）** 是常用的。
>
> 在 3NF 下，没有重复属性；而是使用**查找表**、**超类型-子类型的超类型级公共属性携带**和**实体分解为更小的属性分组**等技术。
>
> 对于 DIV-2，应注意避免**隐藏重叠**（hidden overlaps）——即概念之间存在语义重叠但使用不同的实体、属性或域值名称。
