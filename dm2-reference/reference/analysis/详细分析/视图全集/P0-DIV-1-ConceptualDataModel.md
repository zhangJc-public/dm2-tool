# DIV-1: Conceptual Data Model（概念数据模型）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这个架构中涉及哪些**业务信息**？它们之间是什么关系？

DIV-1 是 DoDAF V2.0 **新增模型**，描述架构域的**高层信息概念**和业务规则结构。它回答"对业务重要的信息是什么"，而非"系统如何存储数据"。

核心问题集：
| 问题 | 回答 |
|------|------|
| 业务关注哪些信息实体？ | 信息类（Entity）列表 |
| 这些信息之间有什么关系？ | 关系类型定义 |
| 每个信息有什么属性？ | 属性或特征描述 |
| 与作战视图的数据元素如何对应？ | OV-2/OV-5b 元素映射 |

## 二、核心内容解析

### 2.1 必要性说明（官方 Purpose 翻译）

> DIV-1 是 DoDAF V2.0 中**新增的 DoDAF 模型**，解决**作战架构上的高层信息概念**问题。
>
> DIV-1 用于记录架构的**业务信息需求和结构化的业务过程规则**。它描述与架构信息相关的信息项、它们的属性或特征及其相互关系。

预期用途：
1. **信息需求（Information Requirements）**
2. **信息层级（Information Hierarchy）**

### 2.2 推荐表达形式

| 形式 | 适用场景 | 备注 |
|------|---------|------|
| **实体关系图（ER 图）** | ⭐ 默认形式 | 信息实体 + 关系类型 + 属性 |
| 类图（UML Class Diagram） | 面向对象风格 | 支持继承和多态 |
| 概念模型图 | 高层抽象 | 不涉及实现细节 |

### 2.3 关键数据元素（DM2 映射）

| 架构元素 | 说明 |
|---------|------|
| **Information Entity（信息实体）** | 业务层面的信息类 |
| **Relationship Types（关系类型）** | 实体之间的关联方式 |
| **Attributes on Entities（实体属性）** | 根据需要关联到实体 |
| **Attributes on Relationships（关系属性）** | 根据需要关联到关系 |

### 2.4 完整细节

#### 2.4.1 DIV-1 vs DM2 的关键区分

⚠️ **重要警告**：

> DIV-1 **不应与 DoDAF 元模型混淆**。DoDAF 架构数据类型（即 DoDAF 定义的架构数据元素和 DM2 实体）是 Performer 和 Operational Activity 这类东西。DM2 确实为 DIV-1 这类 DoDAF 模型提供了底层语义规范。**DIV-1 描述的是特定架构描述范围的信息**。

简单说：
- **DM2** = 描述"架构"本身的元模型（Performers, Activities 等）
- **DIV-1** = 用 DM2 的语义框架来建模"业务信息"（目标跟踪、弹道等业务实体）

#### 2.4.2 互操作风险警示

> 通常，不同组织可能使用**相同的实体名称**来表示**非常不同的信息类型**，且内部结构也不同。这种情况可能带来**显著的互操作风险**——信息模型看起来可能兼容（例如都有 Target Track 数据实体），但对 Target Track 的含义有**不同的和不兼容的解释**。

**DIV-1 解决的核心问题正是这种语义歧义。**

#### 2.4.3 适用场景判断

DIV-1 在以下情况下可能是**必需的**：
- 共享信息的语法和语义构成更高程度**信息系统互操作**的基础
- **信息仓库**是业务活动之间以及能力之间**集成和互操作**的基础

#### 示例：导弹防御架构的信息类

如果架构工作是描述导弹防御，一些可能的信息类：
- **Trajectory（弹道）** — 弹体运动轨迹
- **Target（目标）** — 被追踪的对象
- 关系：将 Target 与某条 Trajectory 关联
