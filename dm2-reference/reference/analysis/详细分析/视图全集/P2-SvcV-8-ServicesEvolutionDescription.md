## 二、核心内容要素

### 2.1 演进描述结构

| 要素 | 说明 | 示例 |
|------|------|------|
| **当前基线 (As-Is)** | 现有服务状态 | 单体应用，SOAP 接口 |
| **目标状态 (To-Be)** | 未来服务愿景 | 微服务架构，RESTful API |
| **中间态 (Intermediate)** | 过渡阶段的快照 | 混合模式，API 网关 + 部分拆分 |
| **迁移步骤** | 每个增量操作 | 步骤1: 引入网关; 步骤2: 拆分用户服务... |
| **触发条件/前置依赖** | 开始某步骤的条件 | 步骤2 需要 IDaaS 先上线 |
| **风险与缓解** | 潜在问题和对策 | 接口不兼容 → 版本共存策略 |

## 五、实践指南

### 5.1 适用场景

✅ **强烈推荐**：
- **SOA → 微服务架构转型**
- **单体应用拆分/重构**
- **遗留系统集成（ESB/API 网关引入）**
- **多云/混合云迁移**
- **DevOps 流水线升级**

❌ **可能跳过**：
- 全新绿地项目（无历史包袱）
- 服务架构简单（<5 个服务）

### 5.2 制作要点

1. **先明确 As-Is 和 To-Be**（SvcV-2 是最好的起点）
2. **每一步都应是可回滚的**——演进不是单向的
3. **关注接口兼容性**——新旧服务共存的根本挑战
4. **与 PV-2 对齐**——演进步骤应映射为项目/子项目
5. **标注每个阶段的技术债务**

## 六、中国适配要点

| 中国背景 | 适配说明 |
|---------|---------|
| **信创替代** | 从国外中间件/数据库迁移到国产品牌时，SvcV-8 描述的是**服务层**的平滑过渡方案 |
| **等保/密评整改** | 安全加固往往需要分步实施，SvcV-8 规划加密服务/认证服务的引入节奏 |
| **政务云迁移** | 从本地数据中心迁移到政务云，服务接口地址/协议的变化路径 |
| **国产数据库替代** | Oracle → 达梦/人大金仓 时 SQL 层面的服务兼容性演进 |

💡 **中国特色场景**: 信创替代中的"双轨运行期"是 SvcV-8 经典应用——新老系统并行运行一段时间，逐步切换流量。
