# DIV-3: Physical Data Model（物理数据模型）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这些数据在系统/服务层面**实际上是如何存储和交换的？**

DIV-3 是 DoDAF 中**最接近实际系统设计的模型之一**——描述逻辑数据模型在系统/服务中的具体实现方式。

核心问题集：
| 问题 | 回答 |
|------|------|
| 系统间交换什么数据元素？ | 物理数据格式 |
| 数据结构是什么？ | 表/消息/文件定义 |
| 如何减少互操作错误风险？ | 精确到字段级的数据描述 |

## 二、核心内容解析

### 2.1 必要性说明

> DIV-3 定义了架构描述中各种系统或服务所使用的系统或服务数据的结构。**物理模式是 DoDAF 中最接近实际系统设计的模型之一**。
>
> DIV-3 用于描述 DIV-2 逻辑数据模型中的信息**如何实际被实现**。

预期用途：
1. **指定系统/服务之间交换的系统/服务数据元素** → 降低互操作错误风险
2. **定义物理数据结构**
3. **提供系统间交换的数据元素的尽可能详细的描述** → 降低互操作问题风险
4. **为系统设计过程提供数据结构**（如需要）
5. **提供数据实现元素的公共字典**（如关系数据库模式的表和记录）→ 在包含物理级数据元素的描述中一致地表达模型
6. **提供系统或服务数据元素的尽可能详细描述** → 降低接口错误风险

### 2.2 推荐表达形式

| 实现技术 | 推荐方法 |
|---------|---------|
| **关系型数据库（RDBMS）** | ER 风格的物理数据库设计图 |
| **面向对象仓库** | 类图 / 对象图 |
| **消息导向** | 引用消息格式标准 |
| **文件传递** | 文件格式描述 |

⚠️ 注意：互操作系统可能使用多种技术交换系统数据，DIV-3 可能有**多个不同分区**，每个分区使用不同的形式。

### 2.3 关键数据元素（SV/SvcV 映射）

DIV-3 中的实体直接对应系统/服务视点的资源元素：

| DIV-3 实体来源 | 对应的视图元素 |
|---------------|--------------|
| **SV-4** Systems Functionality Description | System Resource Flows（系统资源流） |
| **SV-6** Systems Resource Flow Matrix + **SV-10c** Event-Trace | System Resource Elements（系统资源元素） |
| **SvcV-4** Services Functionality Description | Service Resource Flows（服务资源流） |
| **SvcV-6** Services Resource Flow Matrix + **SvcV-10c** Event-Trace | Service Resource Elements（服务资源元素） |
| **SV-10b** / **SvcV-10b** State Transition | Triggering Events（触发事件） |
| **SV-10c** / **SvcV-10c** Event-Trace | Events（事件） |
| **StdV-1** Standards Profile / **StdV-2** Standards Forecast | 因标准要求需要的元素 |

### 2.4 完整细节

#### 2.4.1 Information vs Data 的官方区分

> DoDAF 在作战视点（Operational Viewpoint）中讨论 **Information（信息）**，在系统视点（System Viewpoint）或服务视点（Services Viewpoint）中讨论 **Data（数据）**。
>
> 这种区分的意图是：
> - **DIV-2** 描述对**业务重要**的信息（如条令、SOP 等中引用的信息产品）
> - **DIV-3** 描述在**系统或服务层面相关**的数据

#### 2.4.2 结构化断言（Structural Assertions）

> **结构化断言**——涉及业务规则的静态方面——最好在 DIV-3 中捕获。

这意味着业务规则中关于"数据必须满足的结构约束"应在物理数据模型中体现。

#### 2.4.3 逻辑→物理映射的特点

> 虽然逻辑和物理数据模型之间的映射相对直接，但每个模型的**组件之间的关系**（如逻辑模型中的实体类型 vs 物理模型中的关系表）通常是**一对多或多对多**。

### DIV 三层完整对比

| 维度 | DIV-1 概念 | DIV-2 逻辑 | DIV-3 物理 |
|------|-----------|-----------|-----------|
| **关注点** | 业务信息含义 | 形式化数据结构 | 实现/存储方式 |
| **抽象层级** | 最高（业务语言） | 中间（与技术无关） | 最低（技术特定） |
| **典型用户** | 业务分析师 | 数据架构师 | 系统设计师/DBA |
| **主要产物** | 实体+关系定义 | ER图/UML类图 | 表结构/消息格式/文件规范 |
| **变更频率** | 低 | 中 | 较高（随技术栈变化） |
| **引用来源** | OV-2/OV-5b | OV-1/OV-5b/OV-6a/c + 标准 | SV/SvcV 资源流 + 标准 |
| **规则约束** | 业务语义 | 数据完整性 | 物理存储约束 |
