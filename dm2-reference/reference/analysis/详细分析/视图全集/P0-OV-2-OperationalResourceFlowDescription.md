# OV-2: Operational Resource Flow Description（作战资源流描述）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：作战活动之间需要交换**什么资源**？在哪里需要交换？

OV-2 是作战视点的**核心交互模型**——定义能力需求在作战上下文中的表达方式，展示逻辑层面的资源流动模式。

核心问题集：
| 问题 | 回答 |
|------|------|
| 哪些节点之间需要交换资源？ | Needline（需求线）网络 |
| 需要交换什么类型的资源？ | 信息 / 资金 / 人员 / 物资 |
| 资源流的逻辑模式是什么？ | 不绑定具体系统或位置 |
| "现状"和"目标"有什么差异？ | 资源流（或不流）的差异可视化 |


## 二、核心内容解析

### 2.1 必要性说明

> OV-2 将**作战能力的上下文**应用于预期用户社区。OV-2 的主要目的是**在作战上下文中定义能力需求**。OV-2 也可用于表达**能力边界**。
>
> ⭐ **V2.0 新特性**: 除了信息之外，OV-2 还可用于展示**资金、人员和物资的流动**。

预期用途：
1. 作战概念定义
2. 能力需求细化
3. 协作需求定义
4. 为能力施加本地上下文
5. 问题空间定义
6. 作战规划
7. 供应链分析
8. 活动向资源的分配

### 2.2 核心概念：Needline（需求线）

#### 定义

> **Needline** 文档化所需或实际的资源交换。Needline 是一个或多个资源交换的**通道（conduit）**——即它代表 Resource Flows 的**逻辑束**。
>
> ⚠️ Needline **不指示传输如何被实现**。

#### 关键特征

| 特征 | 说明 |
|------|------|
| **方向性** | 可为单向 |
| **标识符** | 图内唯一 ID + 描述性短语 |
| **逻辑性** | 不涉及物理路由（如中间节点不在图中显示） |
| **聚合性** | 一个 Needline → 多个 Resource Flow（1:N） |
| **场景敏感性** | 不同场景/使命阶段可能显示 Needline 的不同子集 |

#### Needline vs 物理连接的重要区分

> 如果信息在 A 点产生，经 B 点路由，在 C 点使用——**B 点不会显示在 OV-2 上**。Needline 从 A 直达 C。
>
> OV-2 **不是通信链路或通信网络图**，而是资源交换需求的**高层逻辑定义**。

#### Needline 的图形表示

- 用**箭头**表示流向
- 标注**图内唯一标识符**
- 附带描述主要交换类型的短语
- 可用数字+文字标签的组合方式

### 2.3 推荐表达形式

| 形式 | 适用场景 | 备注 |
|------|---------|------|
| **节点-连线图** | ⭐ 默认形式 | 节点=活动/位置，连线=Needline |
| 多层分解图 | 复杂架构 | 按抽象层级或多层级分解 |
| 场景/使命阶段分组 | 大型架构 | 按场景组织多个 OV-2 子图 |

#### 分解方法（官方认可）

1. **多级抽象分解** —— 将 Resource Flow 按层级分解
2. **按活动子集限制** —— 每张图只显示部分活动的 Needline
3. **按场景/使命/阶段组织** —— 最灵活的方式

⚠️ 这些方法可以**组合使用**。

### 2.4 完整细节

#### 2.4.1 四种资源类型

##### （1）信息流（Information）
传统类型，与其他视图紧密关联：
- DIV-1/DIV-2/DIV-3 的数据来源
- OV-3 中详细记录元素属性

##### （2）资金流（Funding）（V2.0 新增）
通过 **Resource Flow Connectors** 表示（区别于 Needlines）

示例：后勤能力中涉及人员实物交付的资金交互

##### （3）人员流（Personnel）（V2.0 新增）
物理意义上的人员移动（非信息意义上的人员记录）

##### （4）物资流（Materiel）（V2.0 新增）
物理物资/装备的流转

示例：空中加油能力中的燃油转移；传感器能力中感知到的物理能量流（非信息流）

#### 2.4.2 Resource Flow Connector vs Needline

| 特征 | Needline | Resource Flow Connector |
|------|----------|----------------------|
| 性质 | **逻辑需求**（需要交换） | **物理上下文**（实际如何交互） |
| 用法 | 表示"必须流动的资源" | 叠加在图上提供业务角色上下文 |
| 视觉区分 | 必须用明显不同的标注 | 与 Needline 区分开 |

#### 2.4.3 OV-2 与 OV-5b 的互补关系

| 维度 | OV-2 | OV-5b |
|------|------|-------|
| **焦点** | Operational Resource Flows（一级关注） | Operational Activities（一级关注） |
| **活动角色** | 次要装饰 | 主要内容 |
| **资源流角色** | 主要内容 | 二级标注（注释或泳道） |

两者通常**共同开发**、迭代进行。

#### 2.4.4 外部资源交互

OV-2 可以定义与**外部资源**的交换需求——即不完全在主题架构范围内但与其接口的：
- 外部 Operational Activities
- 外部 Locations
- 外部 Organizations

这些外部实体可能是：
- 架构内所需的**重要来源**
- 架构所提供产出的**重要目的地**

#### 2.4.5 OV-2 到 SV-1/SvcV-1 的映射

> OV-2 中建立的 Needline 可以由 SV-1 Systems Interface Description 或 SvcV-1 Services Context Description 中的资源和交互来**实现（realize）**。
>
> ⚠️ OV-2 中的一个 Operational Activity + Location 与 SV-1 或 SvcV-1 中的一个资源**可能不是一一对应**的关系：
> - 一个活动可能由两个系统实现（一个备份另一个）
> - 一个活动的功能可能因实际原因拆分到两个位置

| Needline | 需求线 | 资源交换需求的逻辑通道 |
