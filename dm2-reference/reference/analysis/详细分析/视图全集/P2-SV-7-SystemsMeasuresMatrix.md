## 一、视图概述

### 1.1 定义与目的

```
┌──────────────────────────────────────────────┐
│         SV-7: 系统度量矩阵                     │
│     (系统性能参数和质量属性度量)                │
├──────────────────────────────────────────────┤
│                                              │
│  核心问题: "系统的性能表现如何？质量达标吗？"   │
│                                              │
│  ┌──────────┬─────────┬─────────┬──────────┐│
│  │ 性能指标  │ 目标值   │ 当前值   │ 状态     ││
│  ├──────────┼─────────┼─────────┼──────────┤│
│  │ CPU利用率  │ <70%    │ 65%     │ ✅       ││
│  │ 内存占用   │ <80%    │ 78%     │ ✅       ││
│  │ IOPS      │ >5000   │ 5200    │ ✅       ││
│  │ 网络延迟   │ <10ms   │ 15ms    │ ⚠️       ││
│  └──────────┴─────────┴─────────┴──────────┘│
│                                              │
│  用途:                                      │
│  ├─ 系统选型/采购决策依据                     │
│  ├─ 容量规划                                 │
│  ├─ 性能基线建立                             │
│  └─ SLA 底层支撑                             │
│                                              │
└──────────────────────────────────────────────┘
```

**目的**: 描述系统的性能特性和质量属性度量。回答"系统的硬件和软件资源是否满足需求？"

### 1.2 与 SvcV-7 的分工

| 维度 | SV-7 (系统层) | SvcV-7 (服务层) |
|------|-------------|----------------|
| **度量对象** | 服务器、网络设备、存储、操作系统进程 | 服务接口、API 端点、业务功能 |
| **典型指标** | CPU/内存/磁盘/IOPS/带宽/延迟 | 响应时间/吞吐量/可用性/错误率 |
| **关注者** | 基础设施团队/SRE/运维 | 产品经理/业务方/服务所有者 |
| **粒度** | 物理资源级 | 业务逻辑级 |

```
用户请求 → [SvcV-7: API响应时间200ms] → [SV-7: CPU 50% + DB查询5ms]
              ↑ 服务视角                    ↑ 系统视角
           "体验怎么样？"                 "资源够不够？"
```

```
                        ┌──────────────────────┐
                        │     SV-7 度量体系     │

          ┌────────────────────────┼────────────────────────┐

    ┌──────────┐            ┌──────────┐             ┌──────────┐
```

### 2.2 推荐度量项清单

| 类别 | 具体指标 | 单位 | 说明 |
|------|---------|------|------|
| **计算** | CPU 利用率 | % | 平均/峰值/95分位 |
| | 内存使用率 | % | 含缓存/不含缓存 |
| | 进程/线程数 | 个 | 并发处理能力 |
| **存储** | IOPS | 次/秒 | 随机读写能力 |
| | 吞吐量 | MB/s | 顺序读写速度 |
| | 存储使用率 | % | 已用/总量 |
| | 延迟 | ms | 读写响应时间 |
| **网络** | 带宽 | Mbps/Gbps | 链路容量 |
| | 往返延迟(RTT) | ms | 网络质量 |
| | 丢包率 | % | 网络稳定性 |
| | 连接并发数 | 个 | 最大会话数 |
| **安全** | 加解密延迟开销 | ms | 安全模块性能损耗 |
| | TLS 握手时间 | ms | 安全通道建立成本 |
| | 审计日志吞吐 | 条/秒 | 日志写入能力 |

## 三、呈现方式

### 3.1 标准矩阵格式

| 系统/组件 | 指标 | 单位 | 设计目标 | 当前实测 | 状态 | 趋势 | 测量方法 |
|----------|------|------|---------|---------|------|------|---------|
| Web Server | CPU 使用率 | % | ≤70% | 62% | 🟢 | → | Prometheus |
| Web Server | 内存占用 | GB | ≤8 | 6.8 | 🟢 | ↑ | node_exporter |
| App Server | JVM 堆内存 | GB | ≤4 | 3.9 | 🟡 | ↑↑ | JMX |
| Database | QPS | k/s | ≥10 | 11.2 | 🟢 | → | Slow Query Log |
| Database | 查询延迟 P99 | ms | ≤100 | 85 | 🟢 | ↓ | APM |
| Firewall | 规则匹配延迟 | μs | ≤50 | 42 | 🟢 | → | 设备日志 |

### 3.2 AI 增强版指标（现代架构新增）

> 随着 LLM/AI 系统的引入，SV-7 需要覆盖新的性能维度：

| AI 特有指标 | 说明 | 目标参考值 |
|------------|------|-----------|
| **Token 延迟** | 首字输出时间(TTFT) | <500ms |
| **上下文窗口使用率** | 实际 token / 最大 token | <80% |
| **幻觉率** | 事实性错误占比 | <2% |
| **护栏触发频率** | 安全策略拦截比例 | <5%（正常业务） |
| **GPU 显存利用率** | 推理/训练时 GPU 占用 | <90% |
| **推理吞吐量** | tokens/sec 或 req/s | 取决于模型规模 |

## 五、实践指南

### 5.1 适用场景

✅ **建议制作**：
- **基础设施选型/采购**——需要量化对比方案
- **容量规划**——判断是否需要扩容
- **性能调优**——建立优化前后对比基线
- **等保/密评测评**——需要提供安全组件的性能数据

❌ **可跳过**：
- 纯概念设计阶段（无具体技术选型）

### 5.2 制作要点

1. **从 SV-1 系统清单开始**，逐个确定关键指标
2. **区分"硬约束"(必须满足) 和"软目标"(尽力而为)**
3. **使用统计量而非单一瞬时值**——P50/P95/P99 更有意义
4. **定期更新**——性能数据随负载变化而变化
5. **与 SvcV-7 对齐**——确保系统指标能支撑服务级别目标

## 六、中国适配要点

| 中国规范 | SV-7 映射 |
|---------|----------|
| **GB/T 28827 运维标准** | IT 资源监控指标直接适用 |
| **等保 2.0** | 安全设备的性能指标（防火墙吞吐、IDS 延迟）是测评必填项 |
| **信创环境** | 国产化替代后需重新基线化——国产芯片/OS/数据库的性能特征不同 |
| **密评 GM/T 0028** | 密码模块(密码机/卡)的性能指标是专项评测内容 |

⚠️ **信创场景特别注意**: 国产 CPU/OS/数据库在相同配置下的性能参数与国外产品可能存在显著差异，SV-7 在信创项目中必须**基于实测而非厂商宣称值**。
