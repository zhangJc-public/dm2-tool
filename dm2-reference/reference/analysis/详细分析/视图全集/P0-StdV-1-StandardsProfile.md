# StdV-1: Standards Profile（标准配置文件）

## 一、定位与目的

### 1.1 这张图回答什么问题？

**一句话**：这个架构必须遵循哪些**标准、政策和指南**？

StdV-1 是整个架构描述的**技术约束集合**——定义了"能做什么、不能做什么"的规则边界。

核心问题集：
| 问题 | 回答 |
|------|------|
| 哪些技术/操作/业务标准适用？ | 标准清单 + 版本 |
| 每个标准关联到哪个架构元素？ | 标准→元素的映射 |
| 有没有缺少指导的领域？ | 标准空白识别 |
| "现状"和"目标"的标准差异？ | 时间维度对比 |

### 1.2 在架构描述中的角色

```
┌─────────────────────────────────────────────────┐
│           StdV-1: 标准配置文件                    │
│      （技术约束 / 架构设计的规则边界）            │
├─────────────────────────────────────────────────┤
│                                                 │
│   标准类型:                                     │
│   ├─ 技术标准 (Technical Standards)             │
│   │   └─ ISO / 国家标准 / 组织特定标准           │
│   ├─ 操作标准 (Operational Standards)           │
│   ├─ 业务标准 (Business Standards)              │
│   └─ 政策/指南 (Guidance & Policy)               │
│                                                 │
│   关联到以下视图的元素:                          │
│   ├── SV-1/SV-2/SV-4/SV-6 (系统视点)           │
│   ├── SvcV-1/SvcV-2/SvcV-4/SvcV-6 (服务视点)    │
│   └── DIV-2/DIV-3 (数据模型)                   │
│                                                 │
│   ⚠️ 适用(Applicable) ≠ 合规(Conformant)        │
│   └─ 需要额外确认合规程度                        │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 二、核心内容解析

### 2.1 必要性说明

> StdV-1 定义适用于被描述架构的**技术、操作和业务标准、指南和政策**。
>
> DoDAF V2.0 的 StdV-1 还文档化适用于**操作或业务上下文**的政策和标准。
>
> **DISR**（DoD IT Standards Registry）是可用于生成 StdV-1 和 StdV-2 的**技术标准的架构资源**。

预期用途：
1. **标准应用**（为项目策略提供信息）
2. **标准合规性检查**

### 2.2 推荐表达形式

| 形式 | 说明 |
|------|------|
| **标准列表/表格** | ⭐ 默认形式——列出所有适用的标准及其关联元素 |
| 矩阵（标准 × 元素） | 展示多对多关系 |
| 分层配置文件 | 按 As-Is / To-Be / 过渡期组织 |

### 2.3 关键数据元素

| 元素 | 说明 |
|------|------|
| **Standard Reference** | 标准引用（名称+版本+来源） |
| **Applicability** | 适用性——该标准是否适用于某架构元素 |
| **Conformance Level** | 合规程度——需单独评估确认 |
| **Implementation Method** | 实现方式——标准可能允许多种实现方式 |
| **Timescale** | 时间尺度——As-Is / To-Be / 过渡 |

#### 标准关联的架构元素范围

StdV-1 中列出的每个标准都关联到**实现或使用该标准的元素**：

| 视图 | 可关联的元素类型 |
|------|----------------|
| **SV-1** | Systems, System Interfaces |
| **SV-2** | System Resource Flows, Protocols |
| **SV-4** | System Functions |
| **SV-6** | System Resource Flow Matrix 条目 |
| **SvcV-1** | Services, Service Interfaces |
| **SvcV-2** | Service Resource Flows, Protocols |
| **SvcV-4** | Service Functions |
| **SvcV-6** | Service Resource Flow Matrix 条目 |
| **DIV-2** | Logical Data Model 元素 |
| **DIV-3** | Physical Data Model 元素 |

⚠️ 特别注意：**Resource Flow 描述中引用的所有协议都是标准的例子**，无论它们出现在哪个模型中或从哪里引用，都应包含在 StdV-1 中。

### 2.4 完整细节

#### 2.4.1 Applicability vs Conformance（关键区分）

> 与架构元素关联时，**适用性（Applicability）和合规性（Conformance）是有区别的**。
>
> - 如果一个标准**适用于**某个架构，该架构**不需要完全合规**于该标准
> - 对给定标准的合规程度可在每个批准节点基于**风险评估**来判断
>
> ⚠️ 标准与架构元素之间的关联**不应解释为该元素完全符合该标准**。需要进一步细节来确认合规级别。

#### 2.4.2 互操作性的重要补充

> 特定架构的标准配置文件必须与其派生自的**根标准保持完全兼容**。
>
> 此外，StdV-1 模型可以陈述标准的**特定实现方法**，因为**符合标准并不能确保互操作性**。

#### 2.4.3 标准的时间维度

> 每个配置文件都分配一个特定的时间尺度（如"As-Is"、"To-Be"或过渡）。将配置文件链接到定义的时间尺度使配置文件能够考虑：
> - 新兴技术
> - 当前预期更新或过时的技术标准
>
> 如果有多个新兴标准时间段适用于架构，则应同时完成 **StdV-2 Standards Forecast** 和 StdV-1。

#### 2.4.4 构建方法

> 大多数情况下，构建 Standard Profile 包括**识别并列出现有和新兴文档的适用部分**。StdV-1 应识别现有指南以及任何**缺乏指南的领域**。
