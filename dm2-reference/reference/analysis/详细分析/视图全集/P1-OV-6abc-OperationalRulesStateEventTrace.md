# OV-6: 作战行为三件套（Rules + State Transition + Event Trace）

## 概览

OV-6 是作战视点的**行为建模层**，由三个子视图组成：

```
┌─────────────────────────────────────────────────────┐
│                  OV-6 行为建模层                      │
├──────────┬──────────────────┬───────────────────────┤
│  OV-6a   │     OV-6b        │       OV-6c           │
│ 规则模型 │ 状态转换描述      │   事件追踪描述         │
├──────────┼──────────────────┼───────────────────────┤
│ "什么约束"│ "状态如何变化"    │ "按时间顺序发生了什么" │
│ 静态规则  │ 动态状态机        │ 时序交互场景            │
│ 文本为主  │ 图形(状态图)      │ 图形(时序图)           │
└──────────┴──────────────────┴───────────────────────┘

三者互补: 规则定义边界 → 状态机定义合法转换 → 时序图验证具体场景
```

# OV-6a: Operational Rules Model（作战规则模型）

## 一、定位与目的

### 1.1 回答什么问题？

**一句话**：作战/业务活动受哪些规则约束？

OV-6a 是作战视点的**规则库**——捕获所有约束企业业务方式的作战或业务规则。

核心问题集：
| 问题 | 回答 |
|------|------|
| 哪些规则约束了这个作战行动？ | 条令/政策/交战规则 |
| 在什么条件下允许/禁止某操作？ | 命令性/条件性规则 |
| 谁有权在何时执行特定活动？ | 授权规则 |

### 1.2 核心定位

```
OV-6a = OV 系列的 "宪法"
┌────────────────────────────────────────┐
│ 其他 OV 描述结构和运作                   │
│ (OV-1 场景, OV-2 流, OV-5b 活动)        │
│            ↓ 但不充分!                   │
│ OV-6a 补充: 这些结构在什么规则下运作?    │
└────────────────────────────────────────┘
```

> **其他 OV 模型（OV-1、OV-2、OV-5b）大多描述业务的结构和运作，但并不描述其运行的约束和规则。**

## 二、核心内容解析

### 2.1 规则的两种形式 ⭐

DoDAF 明确定义两种规则表达方式：

| 形式 | 结构 | 示例 |
|------|------|------|
| **命令性（Imperative）** | "在所有条件下必须……" | > "战斗损伤评估(BDA) **只应在良好天气条件**下进行。" |
| **条件命令性（Conditional Imperative）** | "如果……则必须……" | > "如果战斗损伤评估显示**打击不完全**，则**应进行再次打击**。" |

### 2.2 规则的层级 ⭐

规范区分了两个层级：

| 层级 | 特点 | 来源 |
|------|------|------|
| **使命级（Mission-level）** | 高层条令性规则 | 条令、指南、交战规则等 |
| **较低层级（Lower levels）** | 架构行为的详细条件规则 | 架构定义过程中推导的条件逻辑 |

较低层级的典型格式：
> **If (这些条件) 存在，且 (此事件) 发生，then (执行这些动作)**

### 2.3 OV-6a vs SV-10a/SvcV-10a 的分工 ⭐⭐

这是一个关键的职责划分：

| 维度 | **OV-6a** | **SV-10a / SvcV-10a** |
|------|-----------|------------------------|
| **关注点** | **What**（做什么约束） | **How**（如何实现约束） |
| **规则类型** | 作战/任务导向的规则 | 资源/系统/服务导向的规则 |
| **抽象层级** | 业务/条令级 | 技术/实现级 |
| **示例** | "BDA 仅在晴天执行" | "加密强度 ≥256 位" |

> **OV-6 是 SV-10 的 What，SV-10 是 OV-6 的 How。**

### 2.4 规则的应用范围

以文本形式定义的 OV-6a 规则可应用于**任何 OV 中定义的架构元素**：
- ✅ 可关联到位置（Locations）
- ✅ 可关联到作战活动（Operational Activities）
- ✅ 可关联到使命（Missions）
- ✅ 可关联到 DIV-2 中的逻辑数据实体
- ✅ 可以叠加显示在 OV-5a 或 OV-5b 上

### 2.5 与 StdV-1 的关系

OV-6a 的规则 ≠ 标准（Standards）本身：
- **OV-6a 规则**：约束使命或架构某些方面的**陈述**
- **StdV-1 标准**：规则的**权威来源和出处**

例如：
- StdV-1 引用 NATO STANAG XYZ 作为标准
- OV-6a 定义："所有目标数据交换**应遵循** STANAG XYZ 格式"

# OV-6b: State Transition Description（状态转换描述）

## 一、定位与目的

### 1.1 回答什么问题？

**一句话**：一个作战活动在不同事件下如何改变其状态？

OV-6b 是作战视点的**动态行为建模器**——用状态机描述活动对事件的响应序列。

核心问题集：
| 问题 | 回答 |
|------|------|
| 这个活动有哪些状态？ | 状态集合 |
| 什么事件触发状态变化？ | 事件→转换映射 |
| 每次转换时执行什么动作？ | 动作定义 |
| 有没有遗漏的状态或死路？ | 完整性检查 |

### 1.2 核心定位

> OV-6b 反映了一个事实：**OV-5a 和 OV-5b 并未完全表达活动响应外部和内部事件时的显式排序**。
>
> 或者说，OV-6b 可以反映单个作战活动内部的显式动作排序或作战活动间的排序。

## 二、核心内容解析

### 2.1 形式基础

> OV-6b 基于 **Statechart Diagram（状态图）**。

状态机的标准定义：

> "一种规范，描述某个动态视图元素的**所有可能行为**。行为被视为对**状态图的遍历**——状态之间通过一条或多条连接的转换弧相连，这些弧由一系列事件实例的分发所触发。在此遍历期间，状态机执行与状态机各元素相关的一系列动作。"

### 2.2 核心要素

| 要素 | 说明 |
|------|------|
| **State（状态）** | 活动在某时刻的条件模式 |
| **Event（事件）** | 触发状态变化的刺激 |
| **Transition（转换）** | 从一个状态到另一个状态的转移 |
| **Action（动作）** | 转换时或状态下执行的行为 |
| **Guard Condition（守卫条件）** | 转换发生的前置条件（可选） |

### 2.3 状态嵌套 ⭐

> **OV-6b 中的状态可以嵌套。**

这使得能够创建相当复杂的模型来表示作战行为。嵌套状态支持：
- 层次化分解复杂行为
- 子状态的复用
- 更清晰的模型组织

### 2.4 与文本规则的双向转换 ⭐

> **状态图可以无歧义地转换为结构化文本规则**，规定作战事件的时间方面和对这些事件的响应，**不会损失含义**。

但图形形式的独特价值在于：
- ✅ 快速分析**规则集的完整性**
- ✅ 检测**死端（dead ends）**或**缺失条件（missing conditions）**

> 如果这些错误在作战分析阶段未能及早发现，往往会导致已部署系统的**严重行为错误**或**昂贵的纠正工作**。

### 2.5 OV-6b vs OV-6c

| 维度 | OV-6b | OV-6c |
|------|-------|-------|
| **视角** | 单个活动的**所有可能行为** | 多个活动的**一次具体场景** |
| **焦点** | 状态空间的全覆盖 | 时间线上的单次执行路径 |
| **类比** | 自动机的状态转换表 | 用例的时序图 |
| **发现的问题类型** | 死状态、不可达状态 | 时序冲突、竞态条件 |

# OV-6c: Event-Trace Description（事件追踪描述）

## 一、定位与目的

### 1.1 回答什么问题？

**一句话**：在一个具体场景中，各个活动按时间顺序如何交互？

OV-6c 是作战视点的**场景演练器**——用时序图追踪关键场景中资源流的有序交换。

核心问题集：
| 问题 | 回答 |
|------|------|
| 这个场景中谁先和谁通信？ | 活动的时序排列 |
| 每个消息包含什么信息？ | 信息内容关联 |
| 所有参与方是否在正确时间获得所需信息？ | 信息可用性验证 |
| 有没有非功能需求被违反？ | 性能约束检查 |

### 1.2 核心定位

> OV-6c 对于从初始作战概念推进到下一层级的细节很有价值。
>
> 它有助于定义**交互和作战线程（Operational Threads）**。

## 二、核心内容解析

### 2.1 作战线程（Operational Thread）⭐

这是 OV-6c 最核心的概念：

> **作战线程被定义为**：一组作战活动，具有活动的**序列和时间属性**，以及完成活动所需的资源。
>
> 一个特定的作战线程可用于描绘军事或**业务能力**。

换句话说：
- **作战线程 = 能力的动态表达**（能力是静态的，线程是其运行时表现）
- 通过对活动和属性建模，能力被定义为完成给定使命目标所需的一组属性

### 2.2 表达形式

OV-6c 也被称为：
- **Sequence Diagrams（时序图）**
- **Event Scenarios（事件场景）**
- **Timing Diagrams（时序图）**

> DoDAF 不认可特定的事件追踪建模方法。OV-6c 可使用**任何支持活动时序和资源流交换布局的建模符号**开发（如 BPMN）。

**不同场景可以用独立的图表表示。**

### 2.3 与其他视图的信息关联

OV-6c 中消息的信息内容可关联到：

| 关联对象 | 视图 |
|---------|------|
| Resource Flows | OV-3 Operational Resource Flow Matrix |
| Resource Flows | OV-5b Operational Activity Model |
| Information Entities | DIV-2 Logical Data Model |

### 2.4 用途

| 用途 | 说明 |
|------|------|
| **作战事件分析** | 分析事件链的合理性 |
| **行为分析** | 验证动态交互的正确性 |
| **非功能用户需求识别** | 发现隐含的性能/安全需求 |
| **作战测试场景** | 为测试提供场景基线 |

## 三、OV-6 三视图协同关系

### 3.1 分工矩阵

| 维度 | OV-6a | OV-6b | OV-6c |
|------|-------|-------|-------|
| **本质** | 约束 | 状态空间 | 执行轨迹 |
| **形式** | 文本规则 | 状态图 | 时序图 |
| **覆盖面** | 全局规则 | 单活动全状态 | 单场景全交互 |
| **静态/动态** | 静态约束 | 动态可能性 | 动态实例 |
| **错误检测** | 规则冲突 | 死状态/缺失条件 | 时序冲突/竞态 |

### 3.2 协同工作流

```
① OV-6a 定义规则边界
   ↓ "在这些规则下..."
② OV-6b 建模所有合法的状态转换
   ↓ "系统可以处于这些状态..."
③ OV-6c 追踪具体的场景执行路径
   ↓ "在这个场景中实际发生了..."
④ 回到 OV-6a: 发现的新约束 → 更新规则
⑤ 回到 OV-6b: 发现的新状态 → 更新状态机
⑥ 回到 OV-6c: 验证修复后的场景 → 确认
```

### 3.3 与下游 SV/SvcV-10 系列的关系

```
作战侧 (OV-6):              实现侧 (SV-10/SvcV-10):
"做什么约束"                 "如何技术性地实施约束"
   ↓ 映射                        ↑ 实现
OV-6a (作战规则) ───────────→ SV-10a / SvcV-10a (系统/服务规则)
OV-6b (状态转换) ──────────→ (对应的状态机实现)
OV-6c (事件追踪) ──────────→ (对应的接口时序验证)
```
