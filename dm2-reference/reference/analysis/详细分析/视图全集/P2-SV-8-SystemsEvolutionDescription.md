## 二、核心内容要素

### 2.1 演进描述结构

| 要素 | 说明 | 示例 |
|------|------|------|
| **As-Is 基线** | 当前系统状态 | 小型机 + Oracle RAC + WebLogic |
| **To-Be 目标** | 未来系统愿景 | x86 集群 + MySQL + Spring Cloud |
| **里程碑节点** | 关键时间点 | FY26Q2/FY27Q1/FY27Q4 |
| **每步动作** | 增量操作 | 引入容器平台、迁移首个子系统... |
| **依赖关系** | 前置条件 | 数据迁移完成后才能下线旧系统 |
| **风险评估** | 潜在问题 | 数据兼容性、人员技能缺口 |

### 2.2 常见演进模式

| 模式 | 适用场景 | 特征 | 风险等级 |
|------|---------|------|---------|
| **渐进式替换** | 一般业务系统 | 逐步替换，新旧共存 | 低 |
| **绞杀者模式(Strangler)** | 单体拆分 | 在前端加路由，逐步蚕食单体 | 中 |
| **双轨运行** | 关键系统(金融/政务) | 新旧并行一段时间，切换流量 | 中 |
| **大爆炸切换** | 非关键系统/全新绿地 | 一次性切换 | 高 |
| **混合云过渡** | 云迁移 | 本地+云端长期共存 | 中高 |

## 五、实践指南

### 5.1 适用场景

✅ **强烈推荐**：
- **IT 现代化/数字化转型**项目
- **遗留系统替换/重构**
- **云迁移/私有云→公有云/混合云**
- **信创替代**（从国外产品迁移到国产）
- **多年度大型项目组合**

❌ **可跳过**：
- 全新绿地项目（无历史包袱）
- 短期一次性开发（<6个月）

### 5.2 制作要点

1. **先画好当前 SV-1 拓扑**——这是 As-Is 的基础
2. **每个步骤必须可回滚**——不要假设一切顺利
3. **数据迁移是最大风险点**——单独标注数据相关动作
4. **与 PV-2 紧密对齐**——演进步骤 ≈ 项目/子项目
5. **标注技术债务**——每步可能产生的新债务
6. **考虑人员因素**——新技术的学习曲线往往被低估

## 六、中国适配要点

| 中国背景 | SV-8 特殊考量 |
|---------|--------------|
| **信创替代工程** | 从 x86+Windows+Oracle → ARM/Kylin+达梦 的完整迁移路线图 |
| **等保整改** | 从不合规架构到三级/四级合规的分步加固计划 |
| **密评改造** | 密码算法(国密SM2/3/4)改造的分阶段实施方案 |
| **政务云迁移** | 从自建机房→政务云平台的系统搬迁计划 |
| **关基保护** | 关键信息基础设施的"三同步"(同步规划/建设/运行)体现为演进节点 |

💡 **信创替代是中国特色最浓的 SV-8 应用场景**：

```
T0: 全外资技术栈 (x86/Oracle/WebSphere)
  ↓ Phase 1: 基础设施替换
T1: 国产硬件 + 国产OS + 外资数据库(过渡)
  ↓ Phase 2: 数据库替换
T2: 国产全栈 - 双轨运行期(新旧并行)
  ↓ Phase 3: 流量切换
T3: 国产全栈 - 主用(外资降级为备用)
  ↓ Phase 4: 全面国产化
T4: 纯国产技术栈 (ARM/达梦/TongWeb)
```
