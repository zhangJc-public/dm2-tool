## Context

DM2 is currently implemented in Python (~4,763 lines, 25 source files) using Typer CLI, Anthropic SDK, and OpenAI SDK. The project uses OpenSpec for its spec-driven change management workflow — OpenSpec is a TypeScript/Node.js CLI tool (`@fission-ai/openspec` v1.3.1) that generates structured artifacts (proposal, specs, design, tasks) and integrates with Claude Code via skill files and slash commands.

The key architectural insight: **DM2 and OpenSpec do not call each other directly**. Claude Code orchestrates both tools via bash — calling `openspec` for workflow management and `python3 -m dm2` for DM2 operations. The two runtimes (Python and Node.js) never share memory, IPC, or direct imports.

### Current State

| Component | Language | Runtime | Role |
|-----------|----------|---------|------|
| DM2 CLI (`dm2`) | Python | CPython 3.x | DoDAF analysis, view generation, knowledge management |
| OpenSpec CLI (`openspec`) | TypeScript | Node.js >=20.19.0 | Change scaffolding, artifact generation, workflow management |
| Claude Code integration | Markdown | N/A (file-based) | `.claude/commands/opsx/` slash commands, `.claude/skills/` skill files |

### The Integration Path (traced)

```
User types /opsx:propose
  → Claude Code reads .claude/commands/opsx/propose.md (generated by OpenSpec)
  → Claude Code calls: openspec new change "<name>"
  → Claude Code calls: openspec status --json
  → Claude Code calls: openspec instructions <artifact> --json
  → Claude Code writes artifact files directly (proposal.md, design.md, tasks.md)
```

At no point does DM2 Python code invoke `openspec`, nor does OpenSpec invoke `dm2`. The "integration" is purely at the Claude Code orchestration level.

### Constraints

- Claude Code is the primary (only) AI coding assistant used with this project
- The project is maintained by a solo engineer
- The DM2 tool must remain installable via `pip install`
- OpenSpec workflow commands (`/opsx:*`) must continue to function
- LLM integration (Anthropic SDK, OpenAI SDK) is core to DM2's view generation capability

## Goals / Non-Goals

**Goals:**
- Determine whether Python is the right long-term language for DM2
- Identify actual (not perceived) bottlenecks at the Python-TypeScript boundary
- Produce a clear recommendation with rationale
- If a migration is warranted, scope the effort

**Non-Goals:**
- Evaluating languages other than Python and TypeScript
- Rewriting DM2 during this evaluation phase
- Changing the DoDAF domain scope or feature set
- Evaluating OpenSpec alternatives (the decision is about DM2's language, not OpenSpec's)

## Decisions

### Decision 1: Keep Python — no rewrite

**Rationale:**

1. **Domain fit**: Python is the dominant language for systems engineering tooling, data modeling, and document generation. DM2's core operations — YAML/Markdown parsing, knowledge indexing, pattern matching — are idiomatic Python with mature standard library and ecosystem support.

2. **LLM/AI ecosystem**: Python has the richest LLM integration ecosystem. The Anthropic SDK and OpenAI SDK are Python-first. Prompt engineering, token management, and streaming are all more natural in Python. A TypeScript rewrite would mean using the JS variants of these SDKs, which lag behind in features and documentation.

3. **No actual language bottleneck exists**: The Python-TypeScript boundary is a mirage. Claude Code calls both tools via shell. There is no shared memory, no IPC, no import dependency. The two tools are independent processes that happen to be used in the same project.

4. **The OpenSpec dependency is deployment-only, not runtime**: OpenSpec generates static Markdown files (`.claude/commands/opsx/*.md`, `.claude/skills/*.md`). Once generated, these files work without OpenSpec being present. A user only needs Node.js when running `openspec update` to refresh skill files after an OpenSpec version bump — not during normal DM2 usage.

5. **Rewrite cost is prohibitive**: ~4,763 lines of working Python with tests, CLI, LLM integration, RAG, knowledge indexing, and cognitive analysis. A TypeScript rewrite would be a multi-week effort with zero user-facing benefit — it would produce the same tool, just in a different language.

**Alternatives considered:**

| Alternative | Why rejected |
|-------------|-------------|
| Rewrite DM2 in TypeScript | Massive effort, loses Python's LLM ecosystem advantage, no user-facing benefit |
| Re-implement OpenSpec in Python | Duplicates effort, diverges from upstream, OpenSpec already works fine via shell |
| Hybrid (Python core + TS shims) | Adds complexity without solving any real problem — there is no integration surface to shim |

### Decision 2: Accept Node.js as an optional dev dependency

**Rationale:** Node.js is only needed for `openspec update` and `openspec init`. Normal DM2 usage (`dm2 init`, `dm2 generate`, `dm2 analyze`, etc.) does not require Node.js. Document this clearly rather than engineering around it.

### Decision 3: Document the architecture

**Rationale:** The confusion that prompted this evaluation ("OpenSpec is TypeScript, DM2 is Python — is this a problem?") is best addressed by clear documentation of how the two tools relate (or don't relate). Add an architecture section to CLAUDE.md explaining the independence of the two tools.

## Risks / Trade-offs

- **[Low] OpenSpec becomes abandonware**: If OpenSpec stops being maintained, the `.claude/commands/opsx/` files are static Markdown and will continue working. Worst case, the project stops using `/opsx:*` commands and manages changes manually. → Mitigation: The generated files are self-contained; no runtime dependency on OpenSpec.
- **[Low] Node.js version requirement drifts**: OpenSpec requires Node >=20.19.0. Future versions may require newer Node. → Mitigation: Node.js is only needed for `openspec update`; pin the OpenSpec version if needed.
- **[None] Python-TypeScript interop issues**: These don't exist because the tools don't interop directly. → No mitigation needed.

## Open Questions

- Should DM2 implement its own native `/opsx:*` equivalent in Python to eliminate the Node.js dependency entirely? (Recommendation: No — the maintenance burden of keeping up with OpenSpec's evolving artifact schema outweighs the trivial cost of `npm install -g @fission-ai/openspec`)
